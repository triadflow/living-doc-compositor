The core reason

The run succeeded because the living doc converted a vague architectural desire into a sequence of verifiable transformations.

False explanation

The work finished because the agent had more momentum, confidence, or willingness to continue.

Actual explanation

The work finished because the objective had become operational: each next action had a direct closure test and a place in the living doc.

The work stopped being "reason about the architecture" and became "make the objective impossible to fake."

The enabling mechanisms

These were the specific conditions that changed the run from drift-prone reasoning into directed execution.

1 · Fixed objective
The objective was no longer negotiable.

The long objective was carried verbatim. That stopped the earlier pattern where a useful template-level refactor could be treated as close enough.

2 · Objective slicing
The tickets followed the words in the objective.

#178-#184 were not arbitrary engineering chunks. Each one corresponded to a phrase: define in code, generate registry JSON, compose templates, preserve runtime, expose MCP access, prevent drift.

3 · Closure gates
Acceptance criteria prevented useful work from becoming fake completion.

After #178 and #179, the living doc still kept the larger criteria open. A slice could close without the objective closing.

4 · Existing machinery
The codebase already had generation paths to attach to.

Template definitions, semantic graphs, template artifacts, registry semantic uses, embed sync, render contracts, and MCP tests already existed. The refactor could connect surfaces instead of inventing a parallel system.

5 · Drift logic
The decisive question became: what should fail if this drifts?

That produced strict registry generation, template graph checks, artifact checks, rendered semantic context checks, MCP checks, and the final cross-surface drift contract.

What made closure real

The key was replacing closure-shaped artifacts with checks that directly conserve the objective.

Closed issues, rendered pages, generated files, and green tests in isolation. Those are useful signals, but they can still miss the objective.

What this teaches about living-doc work

This is the part to carry forward into future skills and repair runs.

Do

Freeze the objective wording before implementing.

The exact wording gives the agent something to conserve. Without it, adjacent useful work can masquerade as completion.

Do

Map tickets to objective phrases.

When every ticket corresponds to a phrase in the objective, sequencing stops being a negotiation.

Do not

Close from artifacts alone.

Rendered pages, commits, and tests are not enough unless they prove the accountable objective terms.

Do

Ask what should fail.

The strongest implementation decisions came from identifying which drift should become impossible to miss.

Do

Update the living doc from implementation evidence.

The living doc became trustworthy because its state changed after source-system proof existed.

Do not

Let a slice become the whole objective.

#178 and #179 were real progress, but the living doc kept the broader criteria open until the source layer, templates, MCP, runtime, and drift checks were all proven.

The general rule

A living doc helps execution when it turns the objective into a conserved system of closure gates. It fails when it becomes a place to narrate progress. The difference is whether a later agent can determine, from the document and the source system, exactly what remains false.
