Internal operating model

Skills for living docs in motion.

A living doc is not merely populated or empty. It passes through recognizable stages while it tries to reach its objective. Those stages are not decorative labels; they emerge from the convergence types present in the document, the cards inside them, and the relationships between those cards.

Core claim

  1. A living doc has an objective.
  2. Its convergence types define the shapes that can carry that objective.
  3. The current state of those shapes reveals the doc's stage.
  4. A useful skill asks stage-specific questions against that structure.
  5. The answer becomes an operation that moves the doc closer to objective-ready.

1. What a Skill Is Assisting

The skill is not assisting a template. It is assisting a concrete living doc as it tries to make something possible.

The template gives a starting structure. The objective gives direction. The convergence types define what kinds of things can be held, compared, proven, decided, tracked, or surfaced. The skill's job is to read the current document state and decide what operation would make the represented thing clearer, more current, more defensible, or more actionable.

The useful unit is:

objective + convergence-type bundle + current relationships -> next useful operation

This is why a skill cannot be only a prompt recipe. It needs stage sense. A proof-ladder section in a fresh claim doc needs evidence discovery. The same proof-ladder in a judgment-stage doc needs confidence discipline and overclaim prevention.

Operational test A skill is valuable when it can answer: "Given this objective and this current structure, what is the next operation that would make the living doc more capable of reaching its objective?"

2. Life Stages of a Living Doc

These are the initial stages. A doc can straddle stages, but one stage usually names the dominant kind of work needed next.

Intent

What is this doc trying to make possible?

The objective exists, but the shape is not yet stable.

  • Clarify objective and success condition.
  • Choose the first convergence-type bundle.
  • Name scope boundaries and exclusions.

Seeding

What first real entities must this doc hold?

The shape exists, but the document is not yet tangible.

  • Find source-backed initial cards.
  • Avoid generic filler.
  • Make the first real source links explicit.

Composition

Where does each thing belong, and what must stay distinct?

Material is entering the doc, but placement and separation are still active work.

  • Classify entities by convergence type.
  • Keep evidence, claim, status, decision, and inference distinct.
  • Preserve source ownership.

Coherence

Do the sections together support the objective?

The doc has material, but the whole structure may not yet hold.

  • Find contradictions and orphan cards.
  • Check whether required relationships exist.
  • Identify missing bridge cards or sections.

Operation

What does this doc make actionable now?

The doc is being used to move the underlying work.

  • Surface blockers and next actions.
  • Update statuses based on evidence.
  • Expose decisions waiting on structure.

Refresh

What moved since the last pass?

The represented world changed or time passed.

  • Add new evidence and source events.
  • Preserve prior state where deltas matter.
  • Resolve stale claims, statuses, and predictions.

Judgment

What can responsibly be decided from this state?

The doc is ready to support a decision, conclusion, or handoff judgment.

  • Separate fact, inference, and decision.
  • Name uncertainty and proof strength.
  • Record rationale and residual risk.

Evolution

Does the document need to change shape?

The current convergence-type bundle no longer fits the represented thing.

  • Add, remove, or split sections.
  • Propose a new convergence type if the registry lacks the shape.
  • Split one doc into multiple docs when the objective bifurcates.

Archive / Handoff

What must remain understandable later?

The doc has served a period, reached closure, or needs another owner.

  • Lock decisions and final state.
  • Preserve canonical links.
  • Expose unresolved residue without reopening the whole doc.

4. Convergence Types as Stage-Defining Shapes

Each convergence type has a local role and a relational role. The stage depends on whether both roles are being carried well enough for the objective.

Local Role

What a section means by itself. A model-assertion holds claims. A proof-ladder holds evidence strength. A decision-record holds decisions. A tooling-surface holds the tools that support the work.

  • Local completeness asks whether the section has valid cards.
  • Local quality asks whether those cards match the type contract.
  • Local freshness asks whether the section reflects current source truth.

Relational Role

What a section must connect to for the whole doc to reach the objective. A proof section only matters if it supports assertions. A tooling section only matters if it supports operations or verification. A citation feed only matters if it grounds positions, indicators, or moves.

  • Relational completeness asks whether required edges exist.
  • Relational quality asks whether the edge is meaningful, not decorative.
  • Relational freshness asks whether linked state still agrees.

5. Relationship Chains Produce Stage Questions

A skill becomes operational when it knows the expected chains for the doc shape and asks stage-specific questions against those chains.

OSS Issue Deep-Dive

Expected chain:

symptom-observation -> code-anchor -> investigation-findings -> attempt-log -> maintainer-stance
  • Seeding: What is the first real symptom, and where is it reported?
  • Composition: Which code anchors plausibly explain the symptom?
  • Coherence: Do findings explain the symptom, or merely describe nearby code?
  • Operation: What attempt should be tried next, and what would count as signal?
  • Judgment: Is there a credible fix path that fits maintainer stance?

Surface Delivery

Expected chain:

status-snapshot -> design-code-spec-flow -> design-implementation-alignment -> verification-checkpoints -> tooling-surface
  • Seeding: What surface, spec, implementation path, and verification path exist?
  • Composition: Which design/spec/code items correspond to the same surface concern?
  • Coherence: Does the implementation actually satisfy the design intent?
  • Operation: Which blocker prevents delivery readiness?
  • Judgment: Is the surface shippable, blocked, or only visually complete?

Proof and Canonicality

Expected chain:

formal-model -> model-assertion -> proof-ladder -> investigation-findings -> decision-record
  • Intent: What truth-bearing thing is being modeled?
  • Composition: Are assertions separated from evidence and decisions?
  • Coherence: Does the proof ladder test the actual assertion?
  • Judgment: What confidence is justified, and what remains unproven?
  • Archive: What decision became canonical, and what evidence backs it?

Competitor Watcher

Expected chain:

competitor-stance-track -> strategic-move-log -> indicator-trace -> proof-ladder -> position-cluster-map -> decision-record
  • Seeding: Which competitors, indicators, and source feeds are in scope?
  • Refresh: What moved this period, and what is only noise?
  • Coherence: Do strategic moves support the claimed posture shift?
  • Judgment: What strategic conclusion is responsible from the evidence?
  • Evolution: Did a new competitor, category, or indicator change the doc shape?

Design System

Expected chain:

taste-signature -> brief-to-system-alignment -> design-system-derivation -> design-system-surface
  • Intent: What client, brief, and taste problem is the system solving?
  • Seeding: What first primitives or surfaces make the system tangible?
  • Composition: Which choices derive from taste, prior work, or brief constraints?
  • Coherence: Does the surface express the claimed taste and answer the brief?
  • Operation: What primitive, pattern, or rule should be built next?

Monitoring Tracker

Expected chain:

expert-stance-track -> citation-feed -> indicator-trace -> proof-ladder -> position-cluster-map -> decision-record
  • Seeding: Which named positions and indicators define the topic?
  • Refresh: What changed since the last period?
  • Coherence: Do citations, indicators, and stance changes agree?
  • Judgment: Which predictions or positions can be resolved?
  • Archive: What was true for this period, and what carries forward?

6. How a Skill Operates on Stages

The skill loop is not generic maintenance. It is an objective-directed pass over a concrete doc structure.

Read the objective and success condition.

The skill starts with what the doc is trying to make possible. If the objective is unclear, the doc is in Intent no matter how many cards it has.

Read the convergence-type bundle.

The section types define the available shapes. The skill identifies which sections are objective-bearing, which are evidence-bearing, which are decision-bearing, and which are support surfaces.

Infer the current stage from structure.

The skill looks at population, evidence, relationships, status freshness, unresolved blockers, and overloaded sections. It does not assume the stage from the template name.

Ask stage-specific structural questions.

In Seeding, ask what real entities are missing. In Coherence, ask whether relationships support the objective. In Judgment, ask what conclusion is justified.

Choose the smallest operation that improves objective reach.

Possible operations include adding cards, linking evidence, lowering a status, recording a decision, splitting an overloaded section, adding a source pointer, or requesting missing context.

Represent the change through the doc's operation channel.

The operation should land as a reviewable patch, MCP operation, or direct source edit according to the environment. The skill should preserve source ownership and avoid inventing canonical truth.

Run the stage completion test.

The skill finishes by naming whether the doc moved stages, stayed in the same stage with a clearer blocker, or needs evolution because the current shape no longer fits.

7. What Skills Need to Contain

A domain skill should be built around stage operation, not around filling fields.

Required Contents

  • Trigger: when this concrete doc shape is being used.
  • Objective reading: what fields establish the doc's purpose.
  • Type bundle map: what each convergence type contributes locally and relationally.
  • Stage detection: how structural signals map to Intent, Seeding, Composition, Coherence, Operation, Refresh, Judgment, Evolution, or Archive.
  • Stage questions: the questions the skill asks at each stage.
  • Allowed operations: what the skill may propose or execute.
  • Completion test: how to tell whether the doc moved closer to objective-ready.

Discipline Rules

  • Do not treat every sparse section as a writing problem; it may be a source discovery problem.
  • Do not treat every populated doc as coherent; relationship quality matters.
  • Do not strengthen status beyond evidence.
  • Do not collapse observed fact, inference, and decision.
  • Do not copy canonical truth when a pointer is the right representation.
  • Do not keep operating inside the old shape when the relationships prove the doc needs evolution.

9. Skill Value by Stage

The skill's value changes with the life stage. Good skills make that explicit.

Early Stages

Intent, Seeding, Composition

  • Prevent vague objectives.
  • Find the first real entities.
  • Place material in the right convergence types.
  • Keep distinctions clean before synthesis hardens.

Middle Stages

Coherence, Operation, Refresh

  • Test whether the doc works as a whole.
  • Surface actionable blockers and next moves.
  • Update current state from source changes.
  • Preserve history where deltas matter.

Late Stages

Judgment, Archive / Handoff

  • Support responsible conclusions.
  • Record decisions and residual uncertainty.
  • Make the doc understandable to a later reader or agent.
  • Prevent last-mile overclaiming.

Shape-Change Stage

Evolution

  • Detect when the current convergence bundle is no longer enough.
  • Split overloaded sections or documents.
  • Propose registry additions only when the missing shape is real.
  • Keep the objective from being trapped in yesterday's structure.

10. Failure Modes This Model Prevents

These are the mistakes that happen when a skill does not understand stage and convergence-type relationships.

Filling Instead of Operating

The skill treats the doc as a form and tries to fill every section. Result: generic prose, weak cards, and no objective movement.

Local Correctness Without Whole-Doc Coherence

Each section looks plausible, but assertions do not connect to proof, findings do not connect to decisions, and tools do not connect to operations.

Stage Confusion

The skill asks judgment-stage questions while the doc is still seeding, or keeps adding source material when the real need is a decision.

Shape Blindness

The skill keeps working inside the current template even when the represented thing has changed enough that the doc needs to evolve.
