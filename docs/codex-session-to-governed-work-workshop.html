Living Docs For Serious Codex Work

A workshop for people who already know how to use Codex. The point is not better prompting. The point is understanding how Living Docs make multi-session Codex work durable enough to continue, inspect, repair, and close.

Audience: experienced Codex users Format: 3-4 hours Focus: Living Docs + lifecycle Output: one usable living-doc work surface

Workshop Objective

Participants understand the Living Doc system: why it helps serious Codex work, how it turns isolated sessions into durable shared state, and how the lifecycle layer extends that state into a repeatable continue/review/repair/close loop.

Starting Assumption

Participants already know Codex can inspect, edit, test, refactor, and explain. The workshop starts where single-session competence stops: continuity, evidence, coordination, handoff, and closure.

Learning Arc

1. Session limitCodex can do work, but the work does not live safely in the session.
2. Living docThe work surface carries objective, state, proof, issues, and next action.
3. Operating rolesWorker, reviewer, repair, continuation, and closure become distinct uses of Codex.
4. LifecycleThe work moves through visible states until it is proven closed or deliberately stopped.
5. HarnessThe harness is automation around a pattern participants can already reason about.
1

The Limit Of Session Competence

Codex can already do meaningful work inside a session. The problem is what happens after the session: what remains true, what remains false, what evidence exists, and what the next session is allowed to trust.

  • The same task gets re-explained.
  • Evidence and final claims drift apart.
  • Follow-up work becomes archaeology.
  • Closure depends on reading the last answer correctly.
Why this matters

The workshop is not about making Codex smarter. It is about making the work durable enough that Codex can re-enter it later.

2

The Living Doc As Work Surface

A living doc is the durable work surface. It is not a final report. It is the place where the objective, acceptance criteria, current state, evidence, blockers, relationship maps, and next action stay visible while the work is still moving.

  • Objective: what must become true.
  • Acceptance criteria: what can pass or fail.
  • State: what is active, repaired, blocked, or ready for review.
  • Evidence: tests, diffs, screenshots, logs, source paths, decisions.
  • Next action: the next valid move.
Participant shift

Before: the session is where the work lives. After: the living doc is where the work lives, and sessions operate on it.

3

Objective-Bound Work

Living Docs make objectives literal. Codex work is judged against the objective and acceptance criteria, not against the confidence of the final answer.

  • Acceptance criteria can fail.
  • Evidence attaches to claims.
  • Open issues remain visible.
  • Progress and closure are separated.
Discussion

Show a piece of Codex work that made progress but should not have closed. Ask what the living doc would need to preserve.

4

Handoff As A First-Class Object

A handoff is not "continue from here." It is a record of what changed, what is proven, what remains false, what must be inspected, and what the next session is allowed to do.

  • It prevents the next session from reconstructing state from chat.
  • It makes uncertainty explicit.
  • It lets review and repair start from evidence.
  • It keeps the human from being the hidden state machine.
Exercise

Take a real Codex task and write the handoff that a fresh session would need to act without re-asking the human.

5

Roles Over Sessions

The same Codex model can act in different roles. The living doc keeps those roles separate so the worker is not the only judge of its own work.

  • Worker: changes source or investigates.
  • Reviewer: judges the objective against evidence.
  • Repair: fixes gaps, drift, weak proof, or stale maps.
  • Continuation: resumes from a known state.
  • Closure review: decides whether done really means done.
Exercise

For one task, assign the next Codex session a role. Decide what the living doc must give that role.

6

The Lifecycle

Once the living doc is the unit, work can move through states instead of collapsing into "done" or "not done."

  • active: work is underway.
  • review-needed: work exists, proof has not been judged.
  • repairable: useful progress, but gaps remain.
  • blocked: outside state is needed.
  • resumable: stopped, but next action is clear.
  • closure-candidate: evidence looks close, but closure still needs review.
  • closed: objective proven.
Key point

The lifecycle is not bureaucracy. It makes the next valid move visible.

7

Reading A Living Doc

Participants learn to inspect a living doc as a working surface, not as prose.

  • Find the objective and success condition.
  • Read acceptance criteria as failure surfaces.
  • Check evidence against claims.
  • Look for blockers, stale maps, weak proof, and unresolved decisions.
  • Identify the next valid Codex role.
Exercise

Read one existing living doc and classify it: active, review-needed, repairable, blocked, resumable, closure-candidate, or closed.

8

Operating A Living Doc With Codex

This is the practical loop participants should leave able to run manually.

  • Start Codex from the living doc, not from memory.
  • Ask Codex to act in one role.
  • Require evidence and state updates.
  • Run a separate review pass when needed.
  • Use the result to continue, repair, block, or close.
Exercise

Run a short worker/reviewer pair against a living-doc task. Compare the result to a normal Codex session.

9

The Harness As The Next Layer

The lifecycle harness automates the pattern participants have just learned manually. It starts bounded runs, captures artifacts, runs reviewer inference, routes repair, and prevents false closure.

  • The deterministic layer controls contracts, artifacts, routing, and terminal state.
  • Inference units do the reasoning inside those contracts.
  • The dashboard shows the operator what happened and why.
  • The harness is understandable because the living-doc pattern came first.
Boundary

Participants do not need to operate the full harness to understand the system. They need to understand the pattern the harness automates.

Participant Output

A living-doc work surface

One task represented as objective, acceptance criteria, state, evidence, blockers, and next action.

A role-based Codex plan

Which Codex session is worker, reviewer, repair, continuation, or closure review.

A lifecycle decision

A justified call: continue, review, repair, block, resume, closure-candidate, or closed.

Useful Operating Prompts

Worker over a living doc
Read the living doc as the work surface. Act only on the stated objective and acceptance criteria. Make the smallest useful change, attach evidence, and update the doc state and next action.
Reviewer over a living doc
Do not edit source files. Review the living doc, changed files, and evidence. Classify the state as closed, closure-candidate, repairable, blocked, resumable, or review-needed. Name the proof and gaps.
Repair over a living doc
Fix only the gaps named by review. Preserve the objective. Update the living doc with changed state, evidence, and any remaining failure surface.
Closure review
Judge whether every accountable term in the objective and acceptance criteria is proven by evidence. If not, classify the state and name the next valid role.

Closing Message

Codex already knows how to work. Living Docs make the work durable enough to continue, inspect, repair, and close. The lifecycle harness is the automated form of that pattern.
