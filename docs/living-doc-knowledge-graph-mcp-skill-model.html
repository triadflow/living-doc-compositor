Internal operating model

Knowledge graph as inference-time substrate.

A living-doc skill should not carry every structural fact inside its prompt. The reusable structure can be represented as a knowledge graph and exposed through MCP, while the skill keeps the domain judgment needed to act in the concrete situation.

Core split

  1. The graph answers what is structurally true, expected, linked, stale, or valid.
  2. The MCP server exposes that graph as inference-time tools.
  3. The skill uses those tools to reason over the current doc state.
  4. The skill still owns domain judgment and responsible action.
  5. The result is a patch, operation, question, or decision that moves the doc toward its objective.

1. What We Are Representing

The knowledge graph is not a second document format. It is the inference-time model of the living-doc system: templates, convergence types, relationships, stages, operations, diagnostics, and prior examples.

The graph should represent the reusable structure that a skill would otherwise repeat in prose. It can say that a surface-delivery doc expects a chain from design-code-spec-flow to design-implementation-alignment to verification-checkpoints. It can say that weak links between those sections are a coherence signal. It can say that stale source timestamps suggest refresh. It can say which operations are valid in that stage.

The graph should not decide that a particular product surface is good enough to ship, that a competitor's move is strategically meaningful, or that a proof is persuasive for a specialized domain. Those decisions require domain judgment, source interpretation, and risk tolerance. They belong in the skill.

living doc JSON + registry + templates + stage model + prior traces -> graph -> MCP tools -> skill reasoning
Boundary principle Put shared structure into the graph. Put situational judgment into the skill. If a fact is true across many docs of the same shape, it probably belongs in the graph. If the answer depends on domain evidence, user intent, or risk, it probably belongs in the skill.

2. Graph Node Types

The graph needs enough semantic granularity for a skill to ask useful questions without loading the whole repo or every skill file.

Document

A concrete living doc instance with objective, success condition, sections, cards, sources, render path, and freshness state.

Template

A starting document shape, such as surface-delivery, competitor-watcher, or design-system.

Convergence Type

A reusable semantic shape from the registry, with status fields, projection, source types, prompt guidance, and local role.

Section

A concrete occurrence of a convergence type inside one document.

Card

A concrete entity inside a section, ideally source-backed and typed by its section contract.

Relationship

An expected or observed edge between types, sections, cards, sources, diagnostics, stages, or operations.

Stage

A lifecycle state inferred from structural signals: Intent, Seeding, Composition, Coherence, Operation, Refresh, Judgment, Evolution, Archive.

Signal

A structural observation such as sparse cards, weak links, stale status, overloaded section, unsupported claim, or source drift.

Operation

A valid action: inspect, add card, link evidence, update status, preserve delta, record decision, split section, propose new type, render.

4. What the Graph Should Hold vs What the Skill Should Hold

The graph is not an attempt to remove skill judgment. It is a way to keep reusable structure out of brittle prompt prose.

Responsibility Split

The knowledge graph supplies structural context. The skill applies domain judgment and chooses a responsible operation.

  subgraph Graph["Knowledge graph / MCP substrate"]
    Templates["Templates"]

    Judgment["Domain judgment"]
    SourceTrust["Source trust"]
    Priority["Question priority"]
    Risk["Risk and overclaim discipline"]
    Stop["When to ask user"]
  end

  Graph -->|queried at inference time| Skill
  Skill -->|chooses| Operation["Concrete operation"]
  Operation --> Patch["Patch / decision / question / source action"]

Graph Holds

  • Template-to-type composition.
  • Expected type relationships.
  • Stage signal definitions.
  • Valid operations by stage.
  • Diagnostics from current doc state.
  • Prior examples and traces.

Skill Holds

  • Domain-specific evidence standards.
  • Objective-ready judgment.
  • Source trust and source interpretation.
  • Question prioritization.
  • Risk, uncertainty, and stop conditions.
  • Voice and output discipline.

Do Not Put in Graph

  • Hard-coded conclusions for a concrete situation.
  • Domain claims that require evidence reading.
  • User-specific preferences masquerading as structure.
  • Prompt text that belongs in a skill.
  • Overfitted lessons from one session.

5. MCP Tool Surface

MCP should expose graph questions as tools. The skill should not need to reconstruct the whole graph by reading files directly.

doc_overview

Return objective, success condition, template, sections, convergence types, source paths, rendered path, freshness metadata.

template_graph

Return expected type bundle, relationship chains, required/optional sections, and known family relationships.

section_graph

Return concrete section/card graph with observed links, source pointers, status fields, and unlinked cards.

stage_diagnostics

Infer stage candidates with evidence: sparse sections, weak links, stale statuses, unsupported claims, overload, decision readiness.

valid_operations

Return operations valid for this doc, stage, and patch contract: add card, update status, link evidence, split section, render.

similar_traces

Return prior sessions, docs, or operations that match the current stage/type/signal pattern.

Inference-Time Query Flow

The skill queries structure first, then applies judgment. The operation is constrained by valid graph/MCP affordances.

7. Prior Sessions as Graph Evidence

Sessions with users can generate examples, but they should enter the graph as traces with provenance and confidence, not as permanent rules by default.

A user session can reveal a new stage signal, a recurring relationship failure, or a useful operation sequence. That should become a trace first. Later, repeated traces can become a graph rule or registry guidance. This keeps the graph from hardening one-off intuition too early.

The useful flow is:

session observation -> trace -> candidate pattern -> repeated support -> graph rule or skill update
Trace discipline A session trace should record what was observed, what structure it involved, what operation was taken, what outcome followed, and whether the lesson is local, tentative, or reusable.

From Session Trace to Reusable Pattern

The graph can learn from work without pretending every session artifact is a universal rule.

9. Anti-Patterns

The graph/MCP approach helps only if the boundary stays sharp.

Graph as a Prompt Dump

If the graph becomes a place to store prose instructions, it loses the advantage of queryable structure.

Skill as a Duplicate Graph

If each skill repeats all expected chains and stage rules, the system drifts and MCP stops being the shared substrate.

Graph as Oracle

If the graph produces domain conclusions instead of structural diagnostics, it will overclaim and hide judgment.

MCP as File Reader Only

If MCP merely returns JSON blobs, skills still have to reconstruct the model manually. The tools should expose semantic queries.

Trace Overfitting

If one good session becomes a permanent rule too soon, future skills inherit accidental behavior.

Judgment Evasion

If the skill hides behind graph diagnostics and avoids making a domain call, the doc does not move toward its objective.

10. Practical Build Order

The graph should grow from real template/doc work, not from an abstract ontology exercise.

Incremental Implementation Path

Start with template graphs and diagnostics for one or two concrete doc families, then generalize only what repeats.

flowchart LR
  Step1["1. Encode current templates as type chains"] --> Step2["2. Add doc_overview and template_graph MCP tools"]
  Step2 --> Step3["3. Add section_graph for observed cards and links"]
  Step3 --> Step4["4. Add stage_diagnostics for structural signals"]
  Step4 --> Step5["5. Add valid_operations by stage and patch contract"]
  Step5 --> Step6["6. Run one concrete domain skill against it"]
  Step6 --> Step7["7. Capture traces from sessions"]
  Step7 --> Step8["8. Promote repeated traces into graph rules"]

Best First Families

  • surface-delivery: clear alignment and verification chain.
  • proof-canonicality: crisp assertion/proof/decision discipline.
  • design-system: distinctive taste/brief/derivation/surface relationship.

First Diagnostics

  • Missing expected section.
  • Sparse objective-bearing section.
  • Unlinked evidence-bearing card.
  • Status stale relative to source timestamp.
  • Decision without supporting proof.

First Operations

  • Suggest source-backed card.
  • Link evidence to claim or checkpoint.
  • Lower or qualify status.
  • Record decision with uncertainty.
  • Ask for missing canonical source.
