The governance layer: turning living docs into a testable substrate

A fingerprint, a skill, and a canvas — and why they matter more than the individual features suggest.

A blog post, not a reference doc · docs/blog-governance-layer.html

Every time an LLM opens your living doc, it silently re-derives the same handful of things: which sections carry which part of the objective, which contracts the doc implicitly follows, whether the current structure still covers the work. That re-derivation is cheap on any single pass. It is expensive in aggregate — two agents reach slightly different conclusions, coverage gaps go unseen until they matter, invariants get enforced only when a human reviewer happens to notice.

This post is about what changes when you stop letting that reasoning happen in the margins and start storing it as data. It is a feature-level narrative — there is a fingerprint, there is a skill, there is a new canvas — but the load-bearing idea is not any one of those. It is that you can turn a living doc from an opinionated piece of structure into a substrate you can actually measure.

What living docs already had

Before we added governance, the compositor already did something sharp. A living doc is a JSON file describing sections. Each section carries a convergence type — a registry-defined shape that says what entities converge into this kind of view: a capability surface, a verification plan, a proof ladder, a decision record. The registry is a vocabulary. New convergence types are new registry entries, not renderer changes.

The type system is the reason agents can reason about a living doc at all. When an LLM sees verification-surface, it knows exactly what fields to expect, what the status set means, and what the section is not for. That is structure doing inference work.

But the type system stops at the boundary of each section. It tells you what a section is. It does not tell you how the sections relate to the doc's purpose, what contracts the doc holds, or whether the objective is still covered. That reasoning has always happened, but it has always happened silently in whatever LLM or human was looking at the doc that day.

Three things agents re-derive silently

If you watch an agent edit a mature living doc, you can see the reasoning work it does without being asked. Three patterns recur.

Coverage. The agent reads the objective sentence and mentally decomposes it into pieces — UI fidelity, repo connect, push runtime. Then it scans the sections to figure out which piece each section addresses. The answer to "is this objective still covered?" lives in the agent's head for exactly one turn.

Invariants. After enough passes, the agent notices that every Capability Map card has at least one code path, that decisions never leak into verification, that model constraints always declare their authorities. These are rules the doc already follows. The agent does not name them; it just keeps following them, which means the next agent has to re-notice them.

Rationale. "Why is this section a formal-model and not a capability-surface?" The agent looks at the content, re-reads the convergence type's structural contract, and concludes. Next agent on the same doc, same question, same answer, same re-derivation cost.

None of this reasoning is wrong. It just leaks out of every conversation. The next pass starts from scratch.

Promoting reasoning to data

The governance layer is a small set of fields that sit at the doc root — next to title, objective, sections — and capture those three kinds of reasoning explicitly:

  • objectiveFacets — the typed decomposition. Each facet has an id, a short name, a one-sentence description.
  • coverage — edges mapping facets to the sections (and specific cards) that carry them.
  • invariants — doc-local contracts, each with an appliesTo scope that's either specific sections or ["*"] (doc-wide).
  • rationale per section — one sentence explaining why the convergence type chosen here was right for this doc's context.

Every field is optional. Existing living docs still render exactly as before. The governance layer is a thing you can opt into when it's useful, and you can opt into it mid-lifecycle. You do not have to commit to it when authoring.

A new convergence type — coherence-map — can render a projection of all of this as its own section inside the doc. Read the doc from the top, you hit the coherence map first, and it tells you in one glance: does every facet have a carrier, are the invariants in play, are any sections orphaned or drifting?

The crystallization moment

There is a skill — /crystallize <doc> — that does the first pass. Read the doc, propose facets from the objective, propose coverage edges from section cards, detect the invariants the doc already follows, write one-sentence rationale per section. The skill's job is to move implicit reasoning into explicit data without changing the doc's content.

But /crystallize is not an authoring gate. You do not run it when you start the doc. You run it once the shape is stable — once you can predict that a week of small edits won't reshape the sections. Running it earlier would calcify a bad decomposition. Running it later is cheap and re-runnable.

Think of the moment as a git commit on the doc's meaning. Before the commit, the doc is still discovering itself. After the commit, the doc has a declared intent that future edits can be checked against.

The fingerprint

Stamped at the bottom of the governance block is metaFingerprint. It looks like sha256:db5226…. It is a hash of the section structure — specifically, each section's id, convergence type, and the ordered list of card ids inside it.

Three things about that choice matter.

It hashes structure, not content. Prose edits inside a card do not change the fingerprint. Adding a card, renaming a section id, changing a convergence type all do. This is deliberate. Governance maps to structure. If you rewrite the text inside a section but keep its shape, your facets and coverage still refer to the same place. If you rename an id, everything referring to that id has a problem, and the fingerprint says so.

It has three states, and each one tells agents how to behave.

StateWhat it meansAgent behavior
MissingNo fingerprint has been stamped. The doc has no declared meaning yet.Fall back to prose re-derivation. Suggest running /crystallize.
FreshStored fingerprint equals current fingerprint. The governance is trustworthy.Deterministically target cards via coverage edges.
StaleStored fingerprint differs from current. Sections have moved since the meta was derived.Refuse to trust coverage. Fall back to prose. Flag loudly.

It makes drift loud. Stale coverage is worse than no coverage, because a stale edge confidently points at the wrong card. The fingerprint mismatch is the signal that flips agents from "trust this data" to "this data is a liar, ignore it until refreshed." Without the fingerprint, the stale state would be invisible.

Stamp vs refresh

Two rituals, different semantics. Stamp just recomputes and stores the hash; it preserves existing facets, coverage, invariants, and rationale. Use it when the structural edit was semantic-neutral (renamed a section you just added, reshuffled a card id).

Refresh via /crystallize <doc> --refresh reruns the skill. It proposes updated facets and coverage for the new structure while preserving the fields the author has edited. Use it when the change is genuinely new shape.

Making it visible: the Flow canvas

A tabular editor is fine for writing governance data. It is bad for seeing it — especially once a doc has ten facets, forty coverage edges, and a handful of invariants.

The compositor gained a new preview tab called Flow that renders governance as a canvas: facets on the left, sections in the middle, invariants on the right, with typed wires between them.

Gray wire — coverage. Facet → section, "is-carried-by." Completeness check.
Dashed red wire — drift. Coverage edge points at a section that no longer exists.
Amber arrow — invariant. Rule → sections, "governs." The actual governance arrow.
Dashed card footer — rationale. Per-section declaration of why this convergence type here.

That visual vocabulary is not decoration. Each color means one thing, each grammar answers one question:

CoverageIs the objective still carried? Gray wires either reach every facet or don't. Orphans show as pink cards with an Orphan badge.
GovernanceWhich contracts does this doc hold? Amber arrows point into sections from the rules that govern them.
Design recordWhy is each section the shape it is? Italicized footer. Dotted stub when missing.

The canvas is also editable. Drag a facet's port to a section's port and you create a coverage edge. Click a wire to remove it. Drag an invariant onto a section and you're governing it. Click the Doc-wide chip to toggle appliesTo: ["*"]. Click a section's rationale card to edit the sentence. When the fingerprint goes stale, the wires dim and a banner offers one-click Stamp or a clipboard-ready /crystallize --refresh prompt.

When the canvas shows your real doc through the governance lens, the abstraction stops being abstract. You see that one section is carrying six facets while another is carrying one. You see which invariants are doc-wide and which apply narrowly. You see at a glance where the reasoning gaps are.

From judgment to verification

Here is the load-bearing claim. Without the fingerprint, "is this doc still coherent?" is always a judgment call. Someone — agent or human — reads the doc, compares it to the objective, and forms an opinion. The opinion takes time to form, varies across people, and decays the moment the context is closed.

With the fingerprint, the same question is a hash comparison. Coverage is a set-difference over explicit edges. Invariant violations are catchable at plan time instead of review time. Drift is a binary state, not a vibe.

That collapse — from judgment to verification — is what turns living docs from opinionated structure into a testable substrate. It means you can do things you could not do before:

  • Measure doc structure over time. Crystallize the same doc after every major revision. Diff the governance. Did facets stabilize? Did coverage get denser? Did invariants accumulate or churn?
  • Compare shapes across docs. Three crystallized docs already produce a usable typology: specify-then-prove docs look different from build-then-verify docs look different from meta-tool docs. The differences are set-theoretic, not impressionistic.
  • Test agent performance empirically. Run an LLM through an edit task with governance present and with it stripped. Measure targeting accuracy, invariant enforcement, cross-agent coherence. The meta layer makes the A/B possible.
The deeper shift

Before governance, you could build a living doc but you could not easily tell whether it was doing its job. After governance, doc quality becomes a series of measurable properties: is every facet carried, do invariants hold, does the fingerprint match, does an agent land on the right card for a given edit?

That is what it takes for living docs to stop being a clever-looking format and start being a component you can reason about inside a larger system.

Try it on your own docs

The governance layer lives in the living-doc-compositor repo. If you have a living doc you maintain:

  1. Clone the repo.
  2. Open your doc in Claude Code and run /crystallize <path/to/doc.json>. The skill proposes facets, coverage, invariants, and rationale from the content you already have.
  3. Render the doc and open it in a browser. Click the pencil icon to enter the compositor, then click the Flow tab. You'll see your governance layer as a canvas.
  4. Pay attention to what surprises you. That's the evaluation signal — the governance layer worked if the canvas shows you something about the doc you didn't already know.

Three real in-repo docs have already been crystallized as dogfood (compositor-development-overview, mobile-companion-workstream, mobile-subscription-refinement-model). Each one revealed at least one structural pattern that hadn't been visible before — a facet carried almost entirely by one section, an invariant that was doc-wide when we thought it was scoped, a coverage fan-out that told us where the real work was landing.

That is what the layer is for. Not to add a new thing to maintain, but to make the doc legible to itself.
