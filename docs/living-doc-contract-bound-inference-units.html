Contract-Bound Inference Units For Living Docs

The current harness shows a general primitive: a bounded inference call receives a contract, inspects living-doc evidence, emits a typed output contract, and hands control back to the lifecycle controller. The useful generalization is not one generic agent. It is a small catalog of typed inference units that operate over a living doc until its objective is reached.

The base primitive is general. The operating unit is specialized. A unit becomes specialized when its prompt, required evidence, permission boundary, and output schema define what kind of reasoning it may perform.

Primitive

Input Contract

Names the living doc, objective, stage, required artifacts, raw log paths, and allowed action surface.

Primitive

Prompt

Turns the generic model call into a role: worker, reviewer, balance scan, repair skill, closure review, or narrative.

Primitive

Inference

Reads the contract and evidence, reasons over the living doc state, and produces a bounded judgment or patch.

Primitive

Output Contract

Returns a typed result the deterministic lifecycle controller can route without pretending to reason.

The Chain Is Held By Contracts

  Doc["Living doc
objective, stage, criteria"] --> Boot["Minimal bootstrap
create initial contract"]
  Boot --> Worker["Worker unit
does objective work"]
  Worker --> Reviewer["Reviewer unit
inspects worker log paths"]
  Reviewer --> Controller["Lifecycle controller
deterministic routing"]
  Controller -->|closed| Stop["Stop: objective proven"]
  Controller -->|user-stopped| UserStop["Stop: user requested"]
  Controller -->|anything else| Next["Next contract-bound unit"]
  Next --> Worker

The controller does not need to understand the objective. Its core rule is intentionally small: closed stops, user-stopped stops, every other signal becomes the next contract-bound inference unit.

What The Contract Must Carry

The contract is the memory surface of the system. It replaces chat continuity with explicit handover.

Living docPath, rendered path, doc hash, current stage, objective, success condition, acceptance criteria.
EvidenceRequired inspection paths, raw worker JSONL paths, previous output-input artifact, proof bundle paths.
Unit typeThe role being invoked: worker, reviewer, balance-scan, repair-skill, closure-review, narrative, or another registered type.
Prompt stanceWhat the unit must optimize for, what it must inspect, what it must not overclaim, and what kind of judgment is valid.
Permission boundaryWhether it may edit source, edit the living doc, run tests, create issues, commit, or only report commit intent.
Output schemaTyped result fields the controller can consume: classification, ordered skills, changed files, commit intent, blocker, closure approval.

Current Specialized Units

The current system already has typed units. They share the primitive but are operationally different because their contracts ask different questions.

Type

Worker

Executes against the living-doc objective. It may change source and/or the living doc according to its run contract.

  • Output: worker result, final message, raw JSONL trace.
  • Risk: claims completion without objective proof.
Type

Reviewer

Inspects raw worker logs and objective evidence, then decides whether closure is proven or another unit is required.

  • Output: stop verdict and next iteration instruction.
  • Risk: accepting progress artifacts as closure.
Type

Balance Scan

Classifies why the living doc is out of balance and orders the repair skill chain.

  • Output: imbalance diagnosis and ordered skill list.
  • Risk: treating a multi-cause imbalance as one simple repair.
Type

Repair Skill

Applies a specialized repair lens to objective, governance, proof, stage, relationship maps, or readiness.

  • Output: repair result, changed files, commit intent, next recommendation.
  • Risk: making local text changes without restoring objective pressure.
Type

Closure Review

Separately checks whether terminal closure is allowed after the reviewer and controller have candidate closure.

  • Output: approved/rejected closure contract.
  • Risk: approving because tests are green while acceptance criteria remain weak.
Type

Run Narrative

Explains a lifecycle run after the fact by reading the artifacts and decisions, not by redoing the work.

  • Output: decision narrative HTML.
  • Risk: quoting messages without showing evidence and changed files.

Repair Skills As Type And Subtypes

Repair is not one thing. It is a family of typed inference units. The balance scan picks subtypes because different imbalances need different reasoning.

flowchart TB
  RepairType["Repair skill type"] --> Obj["Objective conservation
Did we preserve every accountable term?"]
  RepairType --> Energy["Activation energy review
Has proof crossed the real closure threshold?"]
  RepairType --> Equilibrium["Equilibrium rebalance
Did new evidence destabilize the doc?"]
  RepairType --> Path["Reaction path validation
Is the stage transition valid?"]
  RepairType --> Relation["Relationship map alignment
Do diagrams, criteria, and objective agree?"]
  RepairType --> Readiness["Objective execution readiness
Can the next worker execute without drift?"]
  RepairType --> Catalytic["Catalytic repair run
Composed deep repair pass"]

This is why “repair skill” should be treated as a type family. The top-level type says it repairs the living doc state. The subtype says which conservation law, proof threshold, stage path, or relationship surface it is responsible for.

Minimal Bootstrap

The bootstrap should stay small. It should not pretend to solve the objective. Its job is to create the first valid contract so the chain can start.

Create or locate the living doc

The living doc is the substrate. It contains the objective, acceptance criteria, current stage, sections, relationship maps, and proof ladder.

Shape the initial objective contract

The bootstrap checks whether the objective is accountable enough. If not, objective-shaping is the first inference unit.

Start the first worker unit

The worker receives a contract with the living doc path, objective, acceptance criteria, and permitted work surface.

Route from artifacts only

Every later unit is chosen from the previous output contract, reviewer verdict, and living-doc state. No hidden chat memory is required.

A General Initial Catalog

This is the first practical set of inference unit types the system can offer. The list is general, but each type must still have a concrete prompt and output contract.

Foundation

Objective Shaper

Turns a rough session-derived objective into an accountable living-doc objective and acceptance criteria.

Execution

Worker Executor

Performs the work required by the living doc within the current stage and permission boundary.

Evaluation

Evidence Reviewer

Reads raw logs and artifacts to decide closure, repairability, continuation, or user-stop signal.

Routing

Imbalance Classifier

Diagnoses what kind of living-doc imbalance exists and selects the next unit type or repair subtype.

Repair

Governance Repair

Repairs objective conservation, invariants, proof ladder, stage relation, or relationship map alignment.

Closure

Closure Gate

Independently checks whether the objective and acceptance criteria are actually satisfied.

Observation

Run Narrative

Produces a readable explanation of what each unit decided and which contracts caused the next transition.

Externalization

Blocker/Ticket Writer

Creates durable external references when the system needs a traceable blocker, without treating that ticket as closure.

A registry of unit types does not remove intelligence from the system. It gives the controller enough structure to invoke the right kind of intelligence with the right evidence, then route the result without inventing meaning.

The Central Design Rule

Always operate over a living doc

The living doc is not a note beside the work. It is the executable objective surface: objective, stage, acceptance criteria, governance, relationship maps, and proof ladder.

Do not stop on process artifacts

Issues, blockers, failed tests, missing proof, runtime limits, repairability, and former batch limits are not success states. They are inputs to the next contract-bound unit.
