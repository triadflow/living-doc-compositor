Editorial diagnosis

Keep

The core claim is useful: repair skills should rebalance a document against objective, proof, source state, and issue state, not add commentary.

Tighten

The draft names the skill set before showing one concrete drift case. Add a small case so the repair vocabulary has something visible to operate on.

Highest-leverage proposed edits

Opening

Make the failure mode concrete in the first two paragraphs

Current

A living doc can have a strong governance layer and still drift during inference. The problem is not always missing structure. Often the structure is present, but the agent starts treating proof artifacts as permission to close.

Proposed

A living doc can have a strong governance layer and still drift during inference. The common failure is not missing structure. It is false closure: the agent sees green tests, rendered pages, closed tickets, or a plausible proof ladder and starts treating those artifacts as permission to narrow the objective.

Argument
This preserves the point but gives the reader concrete artifacts immediately. It also names "false closure," which is the pressure the rest of the article repairs.
Risk
The phrase "false closure" may become a new project term. Keep it only if you want that concept to recur across repair posts.
Concrete case

Add a short case paragraph before the rule strip

Current
The post moves from the general drift claim directly to repair belonging in skills.
Proposed

The convergence-type refactor is the clean example. Template-generation work produced real artifacts, but the accountable objective was broader: convergence types had to become the executable canonical source, registry JSON had to regenerate from them, MCP had to expose their contracts, and drift tests had to prove the surfaces stayed aligned. A skill that stops at "the generator works" has not conserved the objective.

Argument
The article becomes less abstract if one current repo case demonstrates why balance scan, conservation audit, reaction validation, and activation review exist.
Risk
This ties the post to the current refactor more tightly. That is good for the journal, but it makes the article less timeless.
Repair set

Use canonical skill names in the cards

Proposed

Use exact names consistently: living-doc-balance-scan, objective-conservation-audit, activation-energy-review, equilibrium-rebalance, reaction-path-validator, and catalytic-repair-run.

Argument
The blog is a field journal for a repo. Exact skill names help future agents and readers connect the prose to the actual operating surface.
Risk
Exact names are less elegant in prose. Use them in the card titles, then let the explanatory sentence carry the reader-friendly language.
Boundary

Add a "What this does not solve" section

Current
The article states what repair skills should do, but not what they cannot do.
Proposed

What this does not solve

A repair skill cannot make weak source evidence strong. It cannot close an objective whose accountable terms are still unresolved. It cannot turn a user rejection into a success by adding more explanation. Its job is narrower: make the document honest enough that the next operation is valid.

Argument
This fits the house voice and prevents the skill set from sounding like generic AI magic. It also sharpens the final "honest operating state" claim.
Risk
The article is already short. Add this only if the post should stand alone for readers who have not followed the governance-chemistry work.
Close

Make the final rule more active

Current

The important line is this: a rebalance skill should not end in commentary. It should leave the living doc in a more honest operating state.

Proposed

A rebalance skill should not end in commentary. It should leave the living doc more honest than it found it: objective terms accounted for, invalid closure blocked, and the next operation visible.

Argument
This removes stage-direction phrasing and turns the close into an operational standard.
Risk
Slightly more explicit, slightly less aphoristic. The tradeoff is worth it for this article.

Suggested revised article arc

Pressure: false closure appears even when governance structure exists.

Concrete case: the convergence-type refactor can produce real artifacts while the broader objective remains unresolved.

Distinction: more fields expose pressure; repair skills perform the state transition.

Mechanism: each repair skill checks a different imbalance type before closure, deferral, pivot, or continuation.

Boundary: skills make the document honest; they do not invent missing proof.

Rule: repair leaves the next valid operation visible.
