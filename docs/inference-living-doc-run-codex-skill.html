What The Skill Is For

It gives Codex a controlled way to solve domain-heavy work without losing track of objective, evidence, source ownership, and final readiness.

Purpose
1

Build a governed working doc

Create a fresh objective-specific living doc, or continue an explicitly named run doc, with facets, sections, source links, invariants, and rendered HTML.

2

Use it while doing the work

The doc is not a detached summary. It steers implementation, investigation, verification, trap handling, and handoff.

3

Keep objective completion separate

A coherent document can support completion, but it does not prove the objective is complete. The skill tracks both states explicitly.

Operational Contract

When invoked on an issue, task, ticket, source artifact, or objective, the skill is an execution harness. Codex must attempt to complete the objective in the current run.

Contract

The living doc is working state

It stores the objective frame, source cards, code anchors, attempts, verification evidence, governance, and residual risks.

The living doc is not automatically the deliverable

Unless the objective is itself to create or update a living doc, Codex must still do the actual implementation, investigation, or operational work.

Default commit policy

Checkpoint-commit run doc artifacts at harness gates. Do not commit source-code or source-system changes unless the objective requires it or the user requests it.

Momentum Layer

The recent skill update adds a rule above the harness: every doc or governance action must justify itself by enabling the next objective action.

Momentum
1

Advance the objective

Make source progress first: inspect, implement, test, investigate, update an issue, or reach an evidenced blocker.

2

Update the doc with the smallest useful change

Record only the state or evidence that helps the next step stay coherent.

3

Run the minimum necessary control point

Governance, coverage, render, or checkpoint only as deeply as the uncertainty and risk require.

4

Continue immediately

After a successful checkpoint, move to the next implementation, investigation, or verification action.

Keep the gates. Shorten the gate depth. Force objective action after each gate.

The Run Shape

The full workflow remains available, but the new momentum rules prevent the harness from becoming the work.

Workflow

Scope

Identify objective, success condition, source systems, write policy, and commit mode.

Explore

Read issue facts, linked sources, repo files, tests, docs, history, constraints, and prior attempts.

Structure

Select registry-backed convergence types, facets, invariants, and section rationales.

Create

Scaffold the run doc, add useful source cards, render, govern, and checkpoint.

Work

Implement, investigate, verify, update source systems, and record concrete attempts and evidence.

Reflect

Adjust structure only when discoveries show a missing carrier, wrong type, or governance trap.

Verify

Run tests, source checks, coverage checks, governance checks, and render checks that match the risk.

Finalize

Report readiness, final doc paths, evidence, commits, residual risks, and whether the objective is complete.

Control Points And Budget Rules

The harness keeps strong gates, but now limits pure document activity in non-blocked runs.

Control

Objective Progress Invariant

Every checkpoint must name the next implementation action, investigation action, verification action, or evidenced blocker.

Harness Budget Rule

Do not perform more than one pure-harness cycle in a row unless a governance violation blocks objective work.

Control Point Why It Exists Minimal Valid Version
Coverage Confirms facets have carriers and invalid edges are not hiding drift. Run a quick coverage check and fix only blocking gaps.
Governance Prevents false completion, type-boundary drift, and unsupported status upgrades. Evaluate applicable invariants before status upgrades or source writes.
Render Proves the living doc is visible and standalone. Render after meaningful doc changes or record a render blocker.
Checkpoint Preserves the audit trail and makes the next action explicit. Commit only run-owned doc artifacts unless source commits are in scope.

Two Readiness States

The skill is explicit about the difference between a coherent document and a completed objective.

Readiness

documentReady

The run doc is coherent, rendered, covered, governed, and has no unhandled structural or invariant problems.

  • JSON and HTML exist
  • Coverage has no invalid edges
  • Governance has no unresolved violations

objectiveReady

The actual success condition is satisfied by implementation, investigation, verification, or an evidenced blocker.

  • Work is done or blocked with evidence
  • Tests or source checks support the status
  • Residual risks are explicit

When To Use It

Use the skill when the work benefits from durable, typed state during inference rather than a one-off answer.

Usage

Good Fit

  • GitHub issues with code, tests, and prior context
  • Stuck investigations with multiple source systems
  • Domain-heavy implementation tasks
  • Work that needs explicit verification evidence

Use Minimal Depth

  • Small or well-understood subtasks
  • Low-risk source edits
  • Clear issue scopes with obvious tests
  • Follow-up runs with fresh context already captured

Use Full Depth

  • Ambiguous objectives
  • High-risk source writes
  • State-heavy or cross-system tasks
  • Repeated traps, loops, or false-completion risk
