The Contract

The skill is not a planning style. It is a required execution harness for complex, bounded objectives that need durable state while Codex works.

Core
1

Run the full harness

When invoked, Codex creates a fresh objective-specific governed living doc and uses it throughout the run.

2

Do the source work

The doc supports implementation, investigation, verification, issue updates, PRs, and handoff. It does not replace them.

3

Separate readiness

documentReady and objectiveReady are different states and must be proven with different evidence.

The Momentum Loop

The harness stays useful only when each document action supports the next concrete objective action.

Loop
1

Advance the objective

Inspect, implement, test, investigate, update a source system, or identify an evidenced blocker.

2

Update the doc

Record the smallest useful state or evidence change in the relevant typed section.

3

Run the gate

Use coverage, governance, rendering, or checkpointing where the current risk calls for it.

4

Continue the work

After a checkpoint, move to the named next implementation, investigation, or verification action.

Keep the gates. Shorten the gate depth only when the gate is already satisfied by evidence. Force objective action after each gate.

Control Gates

The gates are strict because they protect the run from false completion, source-boundary mistakes, and status upgrades without evidence.

Governance

Coverage

Objective facets need carriers. Invalid coverage edges are structural problems, not cosmetic warnings.

Governance

Invariants are evaluated before status upgrades, source writes, PR creation, and major phase transitions.

Rendering

Rendered HTML proves the run doc is readable and standalone, while the JSON remains the canonical source.

Delivery Paths

The skill treats source systems as canonical. The living doc summarizes, links, converges, and governs the work.

Output

Code delivery

Objective readiness requires committed source changes, relevant verification, a pushed branch, PR creation or update, PR linkage in the living doc, and focal issue updates when there is an issue.

  • Implementation, tests, fixtures, docs, or verification paths are updated as needed.
  • Commit SHA, PR link, issue update, and verification evidence are recorded.
  • The run doc is checkpointed separately where practical.

Non-code delivery

Objective readiness requires a durable source artifact, verification against source material, explicit confidence limits, and linked residual risks or follow-up tickets.

  • Canonical detail belongs in the source artifact, not pasted wholesale into the run doc.
  • The doc records source links, summary state, and governance evidence.
  • Blocked conclusions must name the blocker and its evidence.

Readiness Split

The final answer must state both readiness values. A polished run doc does not mean the objective is complete.

Status

documentReady

The living doc is coherent, covered, governed, and rendered. JSON and HTML exist, coverage is valid, and no unhandled governance violations remain.

objectiveReady

The actual user objective is complete, or a real evidenced blocker has been reached. Source-system and verification evidence decide this state.

Failure Modes The Skill Guards Against

The governance layer exists because the easiest mistake is to make the document look complete while the real objective is still unfinished.

Traps

False completion

Do not mark implementation, delivery, or migration complete from prose alone.

Source drift

Do not let the doc own operational detail that belongs in GitHub, code, tests, fixtures, or source notes.

Harness loops

Do not perform more than one pure-harness cycle in a row unless governance blocks objective work.

Type blur

Do not mix incompatible convergence types in one section when the registry gives them distinct semantics.
