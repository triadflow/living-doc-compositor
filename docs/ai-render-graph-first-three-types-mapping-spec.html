Living Doc To AI Render Graph / Operational Spec

The missing layer is the exact type-specific mapping from living-doc content into typed AI tasks.

This page specifies that layer for the first three convergence types. It is deliberately document-grounded: the model gets curated content from the exported document plus convergence-type semantics, not internet search, live repo lookups, or external tool calls. The goal is to make implementation straightforward instead of interpretive.

Common Contract

Every profile should be built from the same four payload layers.

The renderer should not hand the model arbitrary HTML or the whole document blob. It should build a small structured payload with stable fields, then let each convergence type add its own normalized item shape.

Doc Root Context

Always include a small document envelope.

  • doc.title
  • doc.scope
  • doc.objective
  • doc.successCondition if present
  • doc.invariants as plain strings if present
  • doc.updated
Section Context

Always include the target section envelope.

  • section.id
  • section.title
  • section.convergenceType
  • section.updated
  • section.callout if present
Type Context

Always include the semantic contract from the registry.

  • type.promptGuidance
  • type.sources
  • type.statusFields
  • type.textFields if present
  • type.detailsFields if present
Normalized Items

Then add a type-specific normalized item list.

The renderer should transform each card in section.data into a small stable JSON shape for that convergence type. That normalized list is what the prompt works over.

Implementation rule: prompt over normalized JSON, not over rendered HTML.
Shared Render Rule

Every type gets the same slot grammar.

Slot id
Bound field
Placement
section-brief
Main summary object for the type
Render above the canonical card grid, clearly labeled as local AI advisory.
section-weakness-note
Highest-risk or weakest-open object
Render as one inline advisory callout between the section heading and the grid.
section-next-loop
Action-oriented next-step object or list
Render below the grid or in the section footer as a collapsible block.
Labeling rule: every advisory slot should say some version of “Local AI advisory” so readers do not confuse it with canonical section content.
Delivery Type design-code-spec-flow

Exact mapping for Design-Code-Spec Flow

This type is about a product surface seen through design artifacts, code refs, spec refs, tracked interactions, and tickets. The AI region should compress surface state and expose the biggest alignment risk.

surface-brief

One short summary of the current product surface and the two or three most important current delivery focus points.

bind to section-brief

alignment-risk-note

One explicit callout naming the surface card where design/code/spec drift appears most dangerous right now.

bind to section-weakness-note

review-checklist

Three to five local review checks a human should walk before sharing or handoff.

bind to section-next-loop
Exact normalized item shape
{
  "name": item.name,
  "feature": item.feature,
  "kind": item.kind,
  "status": item.status,
  "codeStatus": item.codeStatus,
  "updated": item.updated,
  "designRefs": item.pageIds || [],
  "defaultNodeRefs": item.defaultNodeIds || [],
  "codeRefs": item.codeRefs || [],
  "specRefs": item.specRefIds || [],
  "interactionRefs": item.interactionSurfaceIds || [],
  "tickets": item.ticketIds || [],
  "notes": item.notes || []
}

Verification Type verification-surface

Exact mapping for Verification Surface

This type is evidence-rich. The local model should summarize readiness based on explicit coverage, next-step, gaps, automation, API refs, and ticket context, then identify the weakest current signal.

verification-brief

Summarize the actual readiness posture from evidence already present in the section.

bind to section-brief

weakest-signal-note

Identify the probe where the current evidence is weakest and explain why that weakens the readiness claim.

bind to section-weakness-note

next-verification-loop

Produce the next tight verification loop: focus, evidence to collect, and gaps to retire first.

bind to section-next-loop

Proof Type proof-ladder

Exact mapping for Proof Ladder

This type is ordered. The model must reason over rung sequence and current closure boundary, not produce a generic “progress summary.” The first unsatisfied rung is the key analytic output.

proof-state-brief

State what the ladder currently proves, through which rung, and what is still not justified.

bind to section-brief

weakest-open-rung

Identify the first rung that is not actually satisfied and describe the missing evidence for closure.

bind to section-weakness-note

next-stronger-proof

Describe the next evidence shape that would increase proof strength without redesigning the ladder.

bind to section-next-loop
Exact normalized item shape
{
  "order": index + 1,
  "name": item.name,
  "status": item.status,
  "updated": item.updated,
  "tickets": item.ticketIds || [],
  "notes": item.notes || []
}

Default export placement

Put the proof-state brief above the ladder, because it frames the whole sequence. Put the weakest-open-rung callout between the brief and the ladder. Put the next-stronger-proof block at the ladder footer.

This type should never say “the project is progressing well” without naming the actual proof boundary.
Explicit Non-Goals

What this mapping spec does not do

Not web search

No internet lookup is assumed in these prompts. Everything is derived from document content plus type semantics.

Not card mutation

These outputs are advisory overlays in the export. They are not fed back into canonical card fields.

Not renderer freedom

The renderer should not improvise the payload shape per document. It should follow these type-level mappings exactly for the first pass.

Completion bar for this phase: once these three mappings are implemented, the integration can truthfully say it supports type-driven, document-grounded AI export for a meaningful first slice rather than just having a general idea.
