Current Architecture

One canonical model, self-contained HTML outputs

The registry and i18n now live in one authoritative place. The standalone compositor and every rendered living doc remain self-contained, but their embedded data is generated from the canonical JSON files rather than maintained by hand.

Canonical source scripts/living-doc-registry.json and scripts/living-doc-i18n.json
Generated embed docs/living-doc-compositor.html carries generated inline blocks
Guaranteed sync render-living-doc.mjs syncs embeds before rendering

The setup in one sentence

There is one source of truth for type and language data, and there are self-contained HTML artifacts that embed generated copies of that data so they can run anywhere with no server and no external fetches.

single source of truth standalone HTML generated embeds no hand-maintained drift
Why this matters

The system is meant to help AI reason over structured work. If type definitions drift between files, the formal model weakens. If HTML is not self-contained, sharing breaks. This setup keeps both constraints intact at the same time.

Flow of truth

1. Edit canonicals

Change convergence types only in scripts/living-doc-registry.json. Change UI text only in scripts/living-doc-i18n.json.

→
2. Sync standalone embed

scripts/sync-compositor-embeds.mjs rewrites the generated LOCALES and EMBEDDED_REGISTRY blocks inside docs/living-doc-compositor.html.

3. Keep the tool standalone

The compositor still works as one local HTML file because it carries the generated data inline as a fallback when fetch-based loading is unavailable.

→
4. Render any doc

scripts/render-living-doc.mjs runs the sync first, then embeds the current compositor into every rendered living doc.

File Role Edit directly?
scripts/living-doc-registry.json Canonical convergence type, entity type, and status-set model Yes
scripts/living-doc-i18n.json Canonical UI strings for the compositor and rendered docs Yes
scripts/sync-compositor-embeds.mjs Generator that copies canonical data into the standalone HTML Yes
docs/living-doc-compositor.html Standalone compositor GUI with generated embed blocks Yes, except the generated blocks
scripts/render-living-doc.mjs Universal renderer that syncs first, then emits self-contained HTML docs Yes
docs/*.html rendered outputs Shareable living docs with the full compositor embedded No, regenerate them

Generated blocks

The standalone compositor contains two marker-delimited generated regions. They are the only intentional duplication in the system, and they are derived artifacts rather than separate sources of truth. 

/* <generated:locales> */
const LOCALES = { ... };
/* </generated:locales> */

/* <generated:embedded-registry> */
const EMBEDDED_REGISTRY = { ... };
/* </generated:embedded-registry> */

What not to do

  • Do not hand-edit the generated registry or locale blocks inside the HTML.
  • Do not add type knowledge directly into the renderer or compositor logic.
  • Do not patch rendered doc HTMLs by hand when the change belongs in the source JSON or compositor.
  • Do not treat duplicated embedded data as a second canonical definition.

Current command path

These are the commands that matter for keeping the setup healthy.

# Rewrite the generated embed blocks from canonical JSON
node scripts/sync-compositor-embeds.mjs

# Fail if the standalone compositor is out of sync
node scripts/sync-compositor-embeds.mjs --check

# Render a living doc after syncing the embedded compositor data
node scripts/render-living-doc.mjs docs/my-doc.json
Direct file use

docs/living-doc-compositor.html can still be opened directly as a local file. The library fetches may be unavailable, but the tool itself remains complete.

Served use

When served from docs/, the compositor can discover sibling JSON files and act as a document library as well as an editor.

Rendered use

Each rendered living doc remains a fully standalone HTML deliverable with the complete embedded compositor inside it.

AI use

Type semantics stay anchored in one registry, which keeps inference-time contracts stronger than instance-local drift or duplicated schema copies.

The design decision

The system deliberately accepts generated duplication in artifacts so that sharing stays frictionless. What it no longer accepts is semantic duplication in authoring. Canonical semantics live once; generated self-containment can appear many times.
