1. The framing: not all inference is equal

The meta-layer proposal describes a set of structural gains — coverage checks, invariant enforcement, multi-agent coherence, deterministic section targeting. Those gains are real, but their value depends on what the LLM is actually doing on the doc at any given moment.

A doc is touched by two very different kinds of inference work, and they benefit from meta unequally. Recognizing that asymmetry is what makes the second-pass approach attractive rather than a compromise.

2. Reading inference vs. editing inference

Reading

The LLM answers questions, reports state, summarizes sections, surfaces blockers. Structural re-derivation from prose is something LLMs already do well — the gap between "with meta" and "without meta" is small. Mostly you save tokens and tighten consistency across agents.

Editing

The LLM proposes a change — add a fact, move a card, update status, split a section. This is where meta earns itself back: section targeting becomes deterministic, invariants are pre-checked, and coverage gaps surface before an edit rather than after.

The honest split

Without meta, read inference sits at roughly 90% of its realistic ceiling; edit inference sits closer to 60–70%. Most of the gain lives at edit time. That is why shipping meta as a refinement step — rather than as an authoring gate — loses very little while avoiding a real cost at the most creative phase of a doc's life.

3. Why a second-pass skill is the right move

Four reasons to make this a skill you invoke, not a schema the author has to fill in:

Early iteration is shape-discovery

Facets haven't crystallized. Forcing objectiveFacets at authoring time risks calcifying a bad decomposition before the doc knows its own shape.

Meta is a projection of stable structure

You cannot distill what has not settled. A skill pass after the doc's shape is recognizable gets a better decomposition than any template forced upfront.

The skill works from real data

It reads existing sections, extracts candidate facets from prose + titles + card IDs, proposes coverage edges from what is already there, and spots invariants from observed patterns.

Idempotent re-runs solve drift

The same skill becomes the freshness mechanism. Re-run after significant edits; recompute coverage; flag orphaned facets or drifted rationale.

4. The /crystallize skill shape

Call it /crystallize, /coherence-pass, or whatever fits. The shape is the same:

1
Read the doc

Parse objective, sections, and every card's references.

2
Propose objectiveFacets

Decompose the free-form objective sentence into typed facets. Hints from section titles and card IDs.

3
Propose coverage edges

Scan section cards. For each facet, emit the {facetId, sectionId, cardId} tuples where the facet is actually carried.

4
Detect invariants

Observe patterns. If every Capability Map card has codePaths, propose that as an invariant. Don't fabricate rules — surface the ones the doc already follows.

5
Propose per-section rationale

For each section, write one sentence on why this convergence type was picked, derived from the type's structuralContract and the section's content.

6
Write and confirm

Emit doc-root fields + the coherence-map section. Ask the author to accept, tune, or reject the decomposition. Store the accepted version.

7
Stamp a freshness fingerprint

Record a hash of the sections the meta was derived from. The renderer and any downstream agent can later check: is this meta still current with the content it describes?

5. The freshness risk and three mitigations

The second-pass approach has one real weakness. Once edit agents rely on coverage edges for targeting, stale meta is worse than no meta — it confidently points the wrong way. Three mitigations, in increasing strength:

Convention

Pre-flight before editing

Any edit session begins with /crystallize --refresh. Fast, cheap, but it depends on discipline. Works for solo use; fragile in multi-agent settings.

Enforcement

Staleness fingerprint + refuse-to-trust

Meta carries a hash of the sections it was derived from. The renderer and any consuming agent refuse to trust coverage if the hash doesn't match the current sections, forcing a refresh. Failure is loud, not silent. Recommended default.

Integration

Pre-edit hook in the existing /living-doc skill

The bootstrap skill already connects a session to a doc. It gains a step: verify meta freshness, run /crystallize --refresh if stale, then proceed. Invisible to the author in the common case.

The failure mode we are avoiding

Silent drift. An agent looks at stale coverage edges, targets the wrong card for an edit, writes a confident-looking change, and neither the author nor the reviewer notices that the doc's meta stopped matching reality three edits ago. Fingerprints make this noisy instead of silent.

6. What the gain actually is, numerically

Rough order-of-magnitude for agent-assisted work on a mature doc with meta present and fresh:

Section targeting

~75% → ~95%+ Coverage edges point at exact card instead of the LLM guessing.

Invariant breaks

real → near-zero Contracts checked at plan time, not review time.

Multi-agent drift

slow → prevented Rationale + invariants make interpretation structural.

Tokens per edit

meaningful drop Agent skips re-deriving taxonomy on every pass.

Time-to-productive

biggest win A new Codex worker loads meta and knows where things live.

Read-quality

5–15% softer LLMs already re-derive structure well in read mode.

7. Schema implications and backward compatibility

This is a schema evolution, not just a data addition. But the evolution is clean:

  • Every new root field is optional. Docs without objectiveFacets, coverage, or invariants render exactly as today.
  • The coherence-map convergence type is opt-in. A section only appears if the doc declares one.
  • The free-form objective string stays. objectiveFacets is a structured companion, not a replacement. Prose remains human-readable; facets become agent-addressable.
  • Reconciliation rule. If the two drift, the skill treats objectiveFacets as canonical once authored and rewrites the free-form objective as a human-readable restatement on next refresh.
// Doc schema today
{
  objective: string,
  sections: Section[]
}

// Doc schema after adoption (all additions optional)
{
  objective:        string,          // unchanged — stays as human prose
  objectiveFacets?: Facet[],         // structured decomposition (skill-authored)
  coverage?:        CoverageEdge[],  // facet → section/card mappings
  invariants?:      Invariant[],     // doc-local contracts
  metaFingerprint?: string,          // hash of sections at meta-derivation time
  sections: Section[] {
    rationale?: string,              // why this convergence type here
    ... existing fields
  }
}

Clean migration

Every existing doc continues to work untouched. New docs can opt in immediately. Docs that matter most — long-lived references, cross-team workstreams — are the first candidates to crystallize. Early drafts can stay uncrystallized until their shape is recognizable.

8. When to run the skill in a doc's lifecycle

Doc phase Meta state Recommended action
Early authoring — shape unknown None Don't crystallize yet. Let sections settle. Meta would calcify premature choices.
First stable shape recognized None → fresh Run /crystallize once. Author accepts decomposition.
Active editing — small changes Fresh Edit normally. Fingerprint stays valid for minor section edits that don't shift coverage.
Structural change — new section or new facet Fresh → stale Fingerprint invalidates. Re-run /crystallize --refresh.
Archived / deprecated Frozen Leave meta as-is. It becomes historical context.

Mental model

Crystallization is like a git commit on the doc's meaning. You do it when you want a stable snapshot of "here is what this doc is about and how the pieces cover it." Between commits, the fingerprint tells you whether the snapshot is still valid.
