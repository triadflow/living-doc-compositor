Status Snapshot2026-04-17T00:18:02Z

Current state

The repo-backed delivery spine is now proven end to end: a connected repo can install the workflow, write feed events to living-docs-feed, and the phone can sync those events into the app without depending on Apple push.

The current Inbox is a transport and audit surface only. It proves delivery and sync, but it is not the product UX we are aiming for.

The product target has moved to the subscription/refinement model from mockup/subscription-ux: prioritized block-level cards with origin, evidence, intent, grounding warnings, and inline actions such as Watch, Mute, Pin, and Disagree.

Delivery now depends on a small repo-local manifest at .living-docs/manifest.json, while the central library remains the richer cross-system catalog. The two need shared document identity rather than pretending one can replace the other.

The next implementation slice is not better commit-shaped copy. It is grounded event generation and subscription state: richer event schema (#31), user-scoped overlay and subscriptions (#27), the feed/card surface itself (#28, #33, #35), and the refinement-to-package learning loop that lets aggregate user refinements propose package revisions (#32).

The remaining hard gates are real paid-team push verification, disconnect verification against a real repo, and sync hardening such as per-repo cursors or watermarks.

GitHub feed is the delivery source of truth; push is optional acceleration

2026-04-16T14:12:00Z
Ground Truth
Tickets
#22#24#36

The product goal is that living-doc updates appear inside the app itself whenever the app refreshes, even if Apple or Expo push is unavailable.

Connected-repo workflow runs should therefore write a structured GitHub-backed feed that the app syncs into Inbox and the local doc registry.

Expo Push remains useful for waking the user or device, but correctness for in-app visibility cannot depend on paid-team push readiness.

The central living-doc registry should remain the richer cross-system catalog, but commit-time automation cannot depend on a local ~/.gtd file or other central-only state.

Each connected repo therefore needs a minimal repo-local manifest that mirrors delivery-critical fields from the central registry, such as doc id, tracked JSON/HTML paths, title, and public URL.

The central registry remains authoritative for semantic identity and broader system discovery, while the repo-local manifest becomes authoritative for resolving changed files into delivery events during GitHub push workflows.

That repo-local manifest contract now exists in this repo at .living-docs/manifest.json, and the workflow template now consumes it on push.

The current Inbox proves this transport layer, but the newer subscription UX makes clear that the final product surface is a refinement feed of grounded block events, not a raw list of doc-update messages.

Refinements should feed aggregate package revision proposals

2026-04-17T00:18:02Z
Reference
Tickets
#27#32

Per-user refinements from the subscription layer should remain user-scoped at write time, but the system should also aggregate them by role or package to propose revisions to the shared default subscriptions.

Those revisions should surface as reviewable diffs or PRs against subscriptions.json, not as silent automatic mutation of defaults.

The privacy boundary is aggregate only: individual user refinements should not be exposed outside their local overlay.

The unresolved mechanics are sample-size thresholds, handling conflicting signals, and whether any aggregate hints should surface in the mobile UI.

Choose how preview connection state survives refresh

2026-04-16T01:52:30Z
Ground Truth
Tickets
#20#21

Because GitHub secrets are write-only, the app cannot infer real-vs-placeholder token state from the secret value after creation.

The chosen implementation is local metadata in mobile/src/repo-connection-store.ts, keyed by repository full name and storing real or preview plus an ISO timestamp.

This avoids writing an additional readable repo marker while still letting the app upgrade a known preview connection when a real push token becomes available.

If the app is reinstalled or local storage is cleared, GitHub's opaque secret remains unknowable; from a real-token runtime, the row can still Reconnect to overwrite the secret.
