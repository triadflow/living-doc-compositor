A living doc is the substrate, not the report.

The system can keep moving because every inference unit operates over the same living doc: objective, stage, acceptance criteria, proof, and governance.

AI
Codex, with Rene Luijk Triadflow
DraftedMay 10, 2026 Read6 min StatusDraft

A living doc is not just a summary of work after it happens. It is the shared working surface that each model run reads, changes, checks, and hands off to the next run.

In this project, the living doc carries the objective, current state, acceptance criteria, proof, issues, and governance for a piece of work while that work is still moving. A Codex session can read it, change source files, update the document, render the HTML, and leave the next session with more than chat history.

But there was still a leak in the lifecycle: the handoff.

When a session stopped, I was still the one deciding what happened next. I read the last message. I checked whether the proof was real or only closure-shaped. I decided whether the next move should be another worker run, a repair skill, a balance scan, a blocker issue, or a closure review. Then I typed the next instruction.

That is fine for exploration. It is not enough for the system we are trying to build.

I use "substrate" to mean the shared surface that work happens on, not a report produced after the work is finished. The standalone lifecycle harness moves the handoff into the system: it starts bounded model runs, captures their logs and artifacts, asks other runs to review what happened, and writes the next input contract. For that to work, every unit needs to operate over the same conserved state. The living doc is that conserved state.

Terms used in this note

Living doc
The working document that carries objective, state, proof, acceptance criteria, issues, and governance.
Inference unit
One bounded model run with a specific role, such as worker, reviewer, repair, or closure review.
Controller
The small deterministic process that validates contracts and decides which unit runs next.
Contract
The handoff record that tells the next unit what happened, where the evidence lives, and what must be decided next.
Closure
The state where the objective is actually proven, not merely attempted or plausibly progressed.

Earlier notes in this series explain the parts around this idea: a registry gives the document shared terms, governance makes claims testable, repair skills fix drift, and acceptance criteria tell Codex what "done" means. This note focuses on the next problem: what happens when the model run stops.

The controller around the inference units has a small rule. If the objective is proven, it may close. If the user explicitly stops it, it may stop. Everything else becomes the next contract-bound inference unit. A repairable state is not closure. A blocker issue is not closure. A failed proof path is not closure. A runtime limitation is not closure.

The concrete case

The current example is a dashboard run about commitIntent evidence. In plain terms, the goal was to show when a repair unit changed the living doc, prove that the dashboard data came from artifacts, and prove that the lifecycle loop stopped only when the objective was closed or the user explicitly stopped it.

The run is captured in docs/living-doc-harness-commit-intent-dashboard.json.

The leak was the handoff

Before the harness, the living doc could preserve the work, but the transition still depended on a human reading the situation correctly.

The commitIntent run made that visible. The first worker iteration had real progress: dashboard graph code, inspector behavior, artifact-derived commitIntent evidence, and tests. It also had unresolved acceptance criteria. Reviewer inference classified the state as repairable, not closed. That distinction had to survive into the next move.

For example, a worker can produce useful code and still leave one acceptance criterion unproven. In that case, the system should not close the objective. It should preserve the evidence, mark the state as repairable, and route the next unit.

The document had to say that a blocker issue was evidence, not success. It had to keep objectiveReady, currentPhase, the acceptance criteria, and the proof ladder aligned while the lifecycle controller prepared the next input contract. It had to preserve the difference between "work happened" and "the living-doc objective is proven."

If that state lives only in the supervising conversation, the next inference unit reasons from residue. If it appears only in a post-flight report, the repair loop has already made its decisions without it. The substrate has to exist before the next contract is written.

Read the run graph

Read the graph as a lifecycle, not as an architecture diagram. Each card is either a model run or a lifecycle object. Each arrow is a handoff.

The dashboard graph is not a picture drawn after the work. It is a projection of lifecycle artifacts. The labels on the arrows are the control decisions that moved the run forward.

That graph makes the handoff concrete. The controller starts a worker. The worker leaves artifacts. The reviewer reads the worker evidence and returns a verdict. In the first pass, that verdict was repairable, so the next arrow is not "close." It is diagnose repair order.

That decision invokes the balance scan. The balance scan selects a repair path. The repair unit changes the living doc and rendered HTML, then the repair-chain result tells the controller whether the run can resume. Only after that does the second worker iteration run against the repaired substrate.

The living doc appears as its own card because it is not just input text. It is the operated object. Worker and repair units can change it. Review and closure units can judge it. The controller can route from it. The run is not held together by the memory of this chat; it is held together by contracts that keep returning to that document.

The arrow is the contract

This is the small but important shift. In the dashboard, the arrow is clickable because the arrow is carrying evidence. It is not merely a visual connector between two boxes.

In this run, the selected arrow says that the catalytic repair changed the living doc and regenerated the rendered HTML. It also records commit intent: a commit was required, where that requirement came from, why the repair unit was not allowed to commit directly, and which files were changed.

That matters because it separates intelligence from control. The repair unit uses model reasoning to decide what needs to change. The contract records what the unit claims happened. The controller checks the contract and decides where the lifecycle goes next. The controller does not need to understand the whole objective semantically; it only needs enough evidence to route safely.

Why this is substrate

Substrate is the right word because the living doc is the surface every unit can re-enter. The worker reads the objective and changes the system. The reviewer reads the raw JSONL log and the document state. The balance scan reads imbalance. A repair skill changes the doc, proof, issue state, or relationship surface. The closure review checks whether the objective and acceptance criteria are actually conserved.

A report is read after the work.

A substrate is used during the work.

A report explains a completed path.

A substrate helps choose the next path.

Those units are specialized, but the specialization does not make them separate mini-projects. They are all operating over the same living doc. That is what lets the lifecycle continue after a model run stops. The next unit does not inherit a mood. It receives a contract, artifact paths, and the document state it must work against.

This also explains why the controller must remain small. It starts units, validates contracts, and enforces stop rules. It may close only when objective closure is proven. It may stop when the user explicitly stops it. Everything else becomes another contract-bound inference unit.

The next abstraction

Once the living doc is treated as substrate, the next question is not "what should the prompt say?" It is "what kind of unit is allowed to run over this substrate, and what must it prove before the controller trusts it?"

The current system still names too much through prompts. Worker, reviewer, balance scan, repair, closure review, and post-flight summary are already behaving like inference-unit types. The next step is to register those types explicitly: what input contract they accept, what artifacts they must read, what verdicts they may emit, and which next unit types they are allowed to request.

That registry should not replace the living doc. It should make the harness more honest about the units that operate on it. The stable primitive becomes simple: a contract-bound inference unit runs over a living doc, emits artifacts and a verdict, and hands the controller enough evidence to route the next unit.

The boundary

A living doc does not reason by itself. A weak objective is still weak. A relationship map can still imply behavior that the artifacts do not support. A proof ladder can still overclaim. The harness does not make those problems disappear.

What it changes is where those problems surface. They become input for another inference unit instead of becoming silent drift in the supervising conversation. That is the real distinction between a report and a substrate. A report explains what happened after the run. A substrate is what the run keeps using until the objective is genuinely closed.
