Proof Layer for the Living Doc Harness

The agentic harness should not only loop Codex and repair skills. It needs a proof layer: frozen run contracts, state events, sanitized evidence bundles, true blocker semantics, traceable inference logs, and a separate stop-negotiation layer that diagnoses why a run really stopped.

Transferred from TrackAndBack

  • Freeze a run contract before execution starts.
  • Record state transitions as append-only events.
  • Write sanitized evidence bundles for every terminal state.
  • Treat worker stops as raw evidence, not trusted self-report.
  • Show gates, blockers, layers, and trace links in a dashboard.

The Reference Pattern

TrackAndBack already uses a mature version of this idea. The worker does not trust the final model response. It records a run contract, proof state, replay evidence, governance verdict, and dashboard-visible gates.

Run directory

contract.json, state.json, and events.jsonl make the run resumable and inspectable outside chat history.

Evidence bundle

Hashes and summaries preserve proof without exposing raw private material, local paths, secrets, or raw logs.

Dashboard gates

The dashboard derives Record, Source, Proof, Result, Replay, Governance, Evidence, and Blocker posture from files.

Living Doc Harness Contract

The living-doc equivalent should freeze the objective and acceptance surface before each headless run, then record what the model and repair skills changed.

The contract is the durable handoff between inference attempts. If Codex stops, the next action must come from the contract, state, evidence, and skill verdict, not from vague memory of the session.

Stop-Negotiation Inference Layer

The living-doc domain is more open than a parser proof factory. The real reason a model stopped is often implicit in the trace, not stated correctly in the final message.

Do not prompt the worker to produce perfect stop reasons. Let the worker run naturally, then let a separate inference layer diagnose the stop from evidence. 
worker_result = run_codex_headless(contract)

stop_evidence = collect(
  final_message,
  native_trace,
  tool_calls,
  edits,
  proof_results,
  living_doc_before_after
)

stop_verdict = analyze_stop(stop_evidence)

route(stop_verdict):
  true_block       -> write_blocker
  repairable       -> invoke_skill_bundle
  premature_handoff -> resume_with_delta
  closure_candidate -> closure_audit
  proof_failure    -> proof_repair_or_block
  budget_exhausted -> record_remaining_false_terms
Worker role

Execute against the objective. Keep working while a valid next action exists. Do not optimize for explaining why it stopped.

Stop layer role

Read the trace and classify the real stop condition after the run. The final assistant message is only one evidence source.

Dashboard role

Expose the diagnosis, confidence, evidence links, recommended skill, blocker status, and resume recommendation.

Why Not Prompt For Clear Stops?

Clear stop prompting sounds useful, but it can create permission to stop too early. The harness should inspect behavior rather than trust self-classification.

Risky prompt pressure

If the worker is asked to stop cleanly whenever uncertain, it may convert ordinary reasoning difficulty into a user handoff. That makes the harness polite but weak.

Better separation

The worker gets an execution prompt. The stop-negotiation layer gets the diagnostic prompt. The system keeps action and diagnosis separate.

Proof Gates

The dashboard should make the living-doc run legible as gates. Each gate can be done, pending, blocked, residual, or failed.

01Objective

Objective exists, is stable, and has accountable terms.

02Acceptance

Criteria are tied to objective terms and name proof.

03Stage

The current living-doc stage is known and transition is valid.

04Source

Required files, issues, tools, registry, MCP, or external sources are available.

05Stop

Stop reason is classified as repairable, blocked, closure candidate, or budget exhausted.

06Skill repair

Skill invocation produced a concrete verdict and patch or blocker.

07Proof

Commands, artifacts, source state, and rendered doc prove or disprove the claim.

08Closure

Completion is accepted only when objective terms are conserved.

09Evidence

Bundle and summary are written with privacy checks.

10Trace

Native inference logs are available locally and summarized safely.

Dashboard Surface

The dashboard is not just reporting. It is the operator view of the harness contract: where the objective stands, why the run stopped, what skill ran, and whether proof is clean enough to resume or close.

Run board

List living-doc harness runs by objective, stage, stop reason, terminal state, updated time, and active blocker.

Gate strip

Show objective, acceptance, source, skill repair, proof, closure, evidence, and trace status as executable gates.

Evidence drawer

Expose sanitized bundle path, artifact hashes, proof commands, rendered doc path, and privacy flags.

Skill timeline

Show which skill ran after which stop, its verdict, patch summary, and whether it resumed, blocked, or pivoted.

Blocker queue

Separate true blockers from repairable stops, with reason codes, owner layer, required user decision, and issue links.

Trace viewer

Open sanitized trace summaries and local native session log references when a stop or closure claim needs inspection.

True Block Mode

A true block is not a failed loop. It is a controlled terminal state that protects the objective from false progress.

Inference Traceability

The wrapper log is not enough. The harness needs native Codex turn logs and skill invocation logs so a dashboard can inspect why the model stopped, what it tried, and whether the repair was justified.

Native turn logs

Store full local .jsonl event streams for each Codex headless attempt. Keep them local by default.

Sanitized summaries

Extract stop reason, commands run, tool failures, closure claims, and final proposal without raw private payloads.

Trace-to-skill link

Each skill invocation should point to the stop event that caused it and the patch or blocker it produced.

Evidence hashes

Hash logs, rendered docs, contracts, state, and artifacts so the dashboard can prove what it summarized.

Inspection route

When a run blocks or hands off prematurely, the dashboard should tell the operator which trace segment to inspect, not just show the final message.

Closure replay

A closure verdict should be reproducible from the living doc, evidence bundle, proof commands, and trace summary.

Metadata Handovers

Every loop boundary should write a compact handover object. This lets the next inference attempt resume from proof state instead of chat memory.

Run handover

run_id, objective_hash, living_doc_path, stage, current_acceptance_status, source_refs

Stop handover

claimed_stop, inferred_stop_type, confidence, reason_code, turn_log_ref, final_message_hash, attempted_actions

Stop-negotiation handover

verdict, evidence_refs, recommended_skill, resume_prompt_delta, requires_user_decision

Skill handover

skill_name, input_stage, verdict, patch_ref, blocker_ref, resume_recommendation

Proof handover

commands, exit_codes, artifacts, rendered_doc_hash, acceptance_gate_results

Blocker handover

stage, reason_code, owning_layer, required_decision, issue_url, acceptance_to_unblock

Resume handover

next_action, required_context_bundle, unresolved_false_terms, budget_remaining

Resulting Architecture

The living-doc harness becomes a governed proof loop, not just an agent loop.

Without proof layer

Codex runs, stops, summarizes, and waits. The user must infer whether the stop was ambiguity, blocker, proof failure, or premature closure.

With proof layer

The harness records why it stopped, which skill repaired it, what evidence changed, whether closure is valid, and what the dashboard should inspect.
