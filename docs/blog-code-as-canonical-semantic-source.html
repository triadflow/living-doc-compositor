Convergence types as executable semantic source.

A living doc stays JSON because it is state. A convergence type becomes code because it is a reusable semantic contract. Once that contract is executable, the registry, rendered docs, MCP tools, skills, and drift checks can all read from the same source.

Rene Luijk Triadflow
DraftedMay 7, 2026 Read8 min StatusDraft Filedocs/blog-code-as-canonical-semantic-source.html

The system had a quiet mismatch. Convergence types were already the way we named reusable document semantics, but the rest of the stack still had to treat much of that meaning as JSON vocabulary, rendered prose, or template-local convention.

That works until a skill needs to ask a sharper question: what fields does this section own, what status logic is valid, what proof does it expect, and what stage movement would be legal?

The refactor makes that existing convergence-type contract executable in code.

That distinction matters because a living doc is not only a rendered page. It is a structured object that should support reasoning, repair, validation, stage movement, and inference-time tool use. A convergence type is the smallest reusable unit that can carry that pressure. It names a kind of section, its fields, its expected evidence posture, its prompt guidance, its stage behavior, and its contract with other surfaces.

Templates are optional starter compositions, not a mandatory layer. Many living docs will be assembled directly from the set of convergence types a person or skill chooses for the objective at hand. A starter template can suggest that an objective, proof ladder, decision record, implementation slice, acceptance criterion, and risk surface often belong together. But it should not re-own what those types mean, and it should not become the unit of reference for the system.

The concrete change

The source now has a place a reviewer can point at: scripts/living-doc-definitions/convergence-types/. The check is equally concrete: scripts/generate-living-doc-registry-from-definitions.mjs --check --strict. If code definitions, registry JSON, embedded compositor data, template artifacts, rendered context, or MCP output drift apart, the system should fail before a completion claim survives.

The source layer

The source layer now has three different kinds of artifact, and they should not be collapsed into one thing.

Convergence type definitions The reusable semantic contracts. These live in code because they need validation, generation, comparison, contract checks, and direct tool access.
Living doc JSON The concrete document state. This remains portable, inspectable, patchable, renderable, and suitable for handoff.
Registry JSON The generated compatibility and runtime artifact. It gives the compositor and renderer the vocabulary surface they already know how to consume.
Starter templates Optional composed document shapes. They help users start from a useful pattern without becoming mandatory or owning the ground-level semantics.

The practical rule is simple: state stays JSON; reusable semantics become executable; starter templates remain optional compositions instead of becoming the source of truth.

Why this helps inference

A skill working on a living doc needs to know what kind of object it is touching. It needs to know which stage the document is in, which convergence types are active, which relationships matter, which fields are accountable, and what would count as progress toward the objective. If the skill must infer all of that from a long rendered page, it spends reasoning budget rebuilding the substrate.

Code-defined convergence types reduce that burden. The skill can ask an MCP tool for the contract of acceptance-criteria, objective, proof-ladder, or any other type. It can compare the living doc instance to those contracts. It can decide whether the doc is missing proof, overproducing commentary, drifting from objective closure, or mixing stages that should remain distinct.

This is a token benefit, but the deeper benefit is epistemic. The model does not have to invent the rules of the document while editing the document. It can retrieve the rules from the same source that generated the registry, starter-template surfaces, and MCP contracts. That lowers the chance that the skill sounds coherent while operating on the wrong shape.

What the system derives

Once convergence types are executable definitions, the rest of the system can derive more than a list of labels. It can derive the surfaces that humans, renderers, tests, MCP tools, and skills use to stay aligned.

Generated registry The registry becomes an output of code-defined convergence types, not the hand-maintained ground level.
Starter composition Optional templates can assemble convergence-type contracts into useful starting shapes while keeping type-owned meaning in one place.
Template JSON artifacts Starter template artifacts can carry composed semantic metadata without becoming a second source of type definitions.
Relationship graphs Expected section relationships can be generated from the same definitions that drive selected convergence types or starter-template shapes.
Rendered HTML context Standalone rendered docs can include machine-readable semantic context while remaining portable files.
MCP contract access Inference-time tools can ask for convergence-type contracts directly instead of reconstructing them from prose.
Skill reasoning surfaces Skills can load the exact contract, stage pressure, required fields, and prompt guidance relevant to the document they are repairing.
Drift contracts Tests can fail when code definitions, registry JSON, template artifacts, rendered context, or MCP output disagree.
Acceptance proof The refactor can be closed against objective-tied acceptance criteria instead of closed by the existence of plausible artifacts.

This is the value of the move. A single executable semantic source can produce multiple operational surfaces, and those surfaces can be compared. The point is not nicer metadata. The point is fewer silent divergences between what the document means, what the renderer shows, what MCP exposes, what tests verify, and what a skill believes it is operating on.

The living doc stage becomes concrete

Stages are not generic labels pasted on top of a document. A stage is produced by the internal relationship between convergence types. A doc is not "ready" because the word ready appears. It is ready when the objective, scope, invariants, acceptance criteria, source surfaces, proof needs, and open risks are aligned enough that implementation can proceed without guessing.

The same is true for repair. A doc is out of balance when convergence types pull against each other: objectives ask for one closure, tickets implement another, acceptance criteria prove less than the objective requires, or governance invariants name a constraint that the current work path ignores. The stage is the system state created by those relationships.

  1. Fresh The objective exists, but accountable terms, convergence types, acceptance criteria, and source surfaces still need to be made executable.
  2. Executable The convergence types are present, their relationships are coherent, and the objective can drive work without hidden interpretation.
  3. In motion Implementation, evidence, issues, and doc state change together. The living doc must conserve the objective while absorbing new facts.
  4. Out of balance The structure still exists, but the relationships no longer hold. The doc needs repair before more implementation creates misleading closure.
  5. Closed Every accountable objective term is satisfied, blocked, deferred, or explicitly out of scope, and the proof surface agrees with the source system.

Executable convergence-type contracts make those stage judgments less vague. They do not replace judgment. They give judgment a surface to inspect.

What MCP changes

The MCP server becomes more valuable when it exposes contracts that are already canonical elsewhere. A tool call should not be another place where semantic rules are restated. It should be an inference-time doorway into the same convergence-type definitions that generated the registry and template artifacts.

Before a skill edits It can retrieve the relevant convergence-type contracts, expected fields, prompt guidance, and semantic use context.
While a skill repairs It can check whether the repair moves the living doc toward its objective or merely adds closure-shaped material.
Before a claim closes It can compare the objective, acceptance criteria, generated semantic context, source-system proof, and issue state.
After code changes It can detect whether registry JSON, template JSON, rendered context, and MCP behavior still agree with the code source.

This is where smarter inference-time tools become concrete. They are not smarter because they have more prose. They are smarter because they can load the right contract at the right moment, from the right source, and then act against the current living-doc stage.

What this does not solve

Executable convergence-type contracts do not decide the objective. They do not prove that a living doc is current. They do not remove judgment from stage movement or repair. They make the rules inspectable enough that judgment has something stable to inspect.

The acceptance test for the architecture

The architecture is working when a change to a convergence type propagates deterministically across the system. The registry regenerates. Template semantics compose the updated type. Rendered docs carry the updated semantic context. MCP returns the updated contract. Contract tests catch drift if one surface falls behind. Skills can ask for the updated rule at inference time.

That is the closure standard: not the existence of code, JSON, or diagrams, but proof that one executable semantic source produces the operational surfaces the living-doc system depends on, and that those surfaces have not drifted.

The operating rule is tight: a living doc stays JSON because it is state; a convergence type becomes code because it is a reusable semantic contract; optional templates can compose those contracts; MCP exposes them at inference time; tests keep the derived surfaces honest.
