Sharing Model

Sharing living docs without losing lineage

Self-contained HTML makes living docs easy to circulate through WhatsApp, mail, chat, or direct file transfer. That portability is valuable, but it also creates branching copies very quickly. The system needs a clearer distinction between the canonical doc, a shared snapshot, and a local fork.

Canonical Doc The authoritative living doc for a scope, team, or objective.
Portable Snapshot A shareable HTML export for shared reasoning across uneven source access.
Forked Copy A local adaptation that may evolve further and diverge from the original.
Lineage Need Every export must say what it is, where it came from, and how stale it may be.
Portability is good for distribution, but lineage is necessary for coordination.

Without visible identity and provenance, forwarded HTML files stop acting like shared reasoning spaces and start acting like untracked branching artifacts.

The current sharing pattern

One person generates a living doc, exports a self-contained HTML, zips it, and shares it through chat. Team members and management can open the same artifact locally and reason with their own local AI, even when they do not all have the same direct access to GitHub, tickets, or other sources.

Portable Offline-friendly Access-tolerant Easy to forward

Where it turns messy

  • People cannot reliably tell which copy is canonical.
  • Forwarded files look authoritative even when stale.
  • Local AI may reason on different snapshots without anyone noticing.
  • Two near-identical files can represent different source states.
  • Forked local edits are easy to confuse with official snapshots.

Canonical doc

This is the authoritative living doc for a scope. It is the place a team should treat as the current reference surface. A canonical doc may still be rendered to HTML, but the identity and authority of the document must be explicit.

Portable snapshot

This is a transport artifact. It is intentionally easy to share and reason over locally, but it should always describe itself as a snapshot, not silently impersonate the canonical source.

Forked working copy

This is where someone opens the embedded tool, adapts the document, and exports a new branch of the artifact. That is useful, but only if the fork clearly states what it derived from.

Visible export field Why it matters

Quick win now

The fastest improvement is to make every exported HTML visibly carry these identity and lineage fields in the document itself. That does not solve cataloging or publishing yet, but it sharply reduces ambiguity in the current WhatsApp-style sharing pattern.

What this does not require

  • No new permission model.
  • No central publishing system yet.
  • No removal of self-contained HTML export.
  • No reduction of the embedded tool.

What it gives immediately

  • Less confusion between official snapshots and local forks.
  • Clearer reasoning for team members and management.
  • Safer local AI use over shared artifacts.
  • A foundation for later catalog and publishing workflows.

The structural principle

A living doc export should remain portable, but it should never be anonymous. The artifact must say what it is, who stands behind it, where it came from, and how it relates to the canonical reasoning surface.

This keeps portability as a strength without letting circulation turn into uncontrolled document drift.
