Why code representation helps.

The living doc itself should remain a JSON artifact. The value of code is elsewhere: the semantic machinery around living docs has behavior. Code can define that behavior once, then generate the registry, templates, MCP diagnostics, skill context, tests, and diagrams from the same source.

Simple value

  1. JSON is good for stored document state.
  2. Code is better when structure has behavior.
  3. Living-doc semantics have behavior: relationships, stages, diagnostics, operations.
  4. Typed definitions can generate the JSON and the MCP behavior.
  5. The result is less duplicated meaning and less drift.

1. The Problem Is Not JSON

The problem is asking JSON to carry behavior that currently lives elsewhere.

A living doc JSON file is useful because it is portable, reviewable, renderable, patchable, and archival. It should keep being the source of truth for one concrete document instance.

The friction appears around the machinery that interprets that instance. Convergence types, template chains, stage signals, relationship gaps, governance checks, valid operations, MCP diagnostics, and skill prompt context are all behavioral. They do not merely say what data is present. They say what the data means, what should be checked, what stage the doc may be in, and what operation is valid next.

JSON stores state. Code defines behavior over state.
Keep the artifact boundary Do not turn every living doc into source code. Turn the reusable semantic machinery into typed definitions that can generate JSON artifacts and runtime behavior.

3. What Code Representation Adds

Code does not add value because it is more fashionable than JSON. It adds value because it can define reusable behavior, compose definitions, run checks, and generate multiple outputs.

Executable Relationships

Expected chains can become testable rules, not prose.

expects("proof-ladder")
  .toSupport("model-assertion")

Stage Signals

Stage inference can become diagnostics over current document state.

signal("coherence-risk")
  .when(missingEdge("proof", "assertion"))

Valid Operations

Operations can be constrained by stage, type, patch schema, and source ownership.

operation("link-evidence")
  .validIn("coherence", "judgment")

Generated Artifacts

One definition can generate registry JSON, template JSON, MCP schemas, prompt fragments, tests, and diagrams.

Fail-Fast Evolution

Adding or changing a type can fail tests if stage roles, relationships, or operations are missing.

Inference-Time Compression

Skills can receive compact generated context instead of reading the entire registry and hand-authored explanations.

4. Source Code as Semantic Definition

This is the kind of code representation that would help: not arbitrary application code, but a typed semantic definition layer.

defineTemplate("surface-delivery", {
  objectiveRole: "make one product surface deliverable and inspectable",
  sections: [
    "status-snapshot",
    "design-code-spec-flow",
    "design-implementation-alignment",
    "verification-checkpoints",
    "tooling-surface"
  ],
})
  .expects("design-code-spec-flow")
  .toFeed("design-implementation-alignment")
  .expects("design-implementation-alignment")
  .toBeTestedBy("verification-checkpoints")
  .stageSignal("coherence", missingExpectedRelationship())
  .stageSignal("operation", blockerVisibleWithNextAction())
  .operation("add-verification-checkpoint")
  .operation("link-implementation-evidence")
  .operation("lower-delivery-status");

The important part is that this definition is executable. It can validate a template, generate a diagram, answer an MCP query, and produce stage-specific skill context.

The definition is still declarative. It is not hand-writing the whole MCP server. It is making the relationships and stage rules first-class enough that tooling can use them.

7. The OpenUI Lesson, Translated Carefully

OpenUI is not proof that every structure should become code. It shows a useful pattern: when a structure must generate prompts, validate output, render, stream, execute, and update, a passive JSON source becomes limiting.

OpenUI Need

Generated UI must stream, validate, render, execute queries/mutations, support state, and update incrementally.

Their Move

Define component libraries and schemas in code; generate prompts and specs; use a compact language for model output.

Our Translation

Keep living doc JSON as artifact; define registry/template/stage/operation machinery in typed code that generates runtime surfaces.

Parallel Pattern

The relevant pattern is not "JSON bad." It is "behavioral structure benefits from executable definition."

9. The Value in One Example

Without code representation, the relationship between assertion, proof, findings, and decision is described in several places. With code representation, it becomes an executable contract.

defineTemplate("proof-canonicality")
  .sections([
    "status-snapshot",
    "formal-model",
    "model-assertion",
    "proof-ladder",
    "investigation-findings",
    "decision-record",
    "tooling-surface"
  ])
  .expects("formal-model").toFrame("model-assertion")
  .expects("proof-ladder").toSupport("model-assertion")
  .expects("investigation-findings").toInterpret("proof-ladder")
  .expects("decision-record").toBeJustifiedBy("investigation-findings")
  .stageSignal("coherence", missingExpectedRelationship())
  .stageSignal("judgment", allCriticalRelationshipsSupported())
  .operation("weaken-assertion")
  .operation("link-proof-to-assertion")
  .operation("record-decision-with-uncertainty");

From this one definition, MCP can diagnose: "The doc has a model assertion and proof ladder, but no proof card supports the assertion. Likely stage: Coherence. Valid operations: link proof, add proof, or weaken assertion."

The proof-canonicality skill then decides which operation is responsible in the actual domain. The skill does not need to rediscover the structural contract from scratch.

10. Costs and Guardrails

Code representation is not free. It is only worth it if it reduces duplicated behavior without making authoring painful.

Risk: Over-Engineering

A large abstract DSL can become harder to change than JSON. Start with one or two templates and only encode repeated behavior.

Risk: Losing Inspectability

If generated JSON becomes opaque, contributors lose trust. Generated artifacts must remain readable and diffable.

Risk: Behavior Hidden in Code

Definitions should generate docs and diagrams so the behavior stays visible to humans and agents.

Guardrail: JSON Artifacts Remain Canonical for Docs

Concrete living docs stay JSON. Code defines the machinery, not the lived document content.

Guardrail: Round-Trip Tests

Generated registry/templates should pass the existing renderer, compositor embed, MCP, and validation tests.

Guardrail: Progressive Adoption

Begin by generating a template graph or diagnostics module, not replacing the whole registry at once.

11. Practical First Step

The smallest valuable experiment is not a full rewrite. It is an executable relationship definition for one template family.

Minimum Useful Experiment

Pick one template, encode its relationships and stage signals, generate one MCP diagnostic, and compare that against the existing hand-coded reflection output.
