Team Model

Living docs as runtime composition for deep collaborative work

This page brings together the current discussion about moving from a personal living-doc tool toward a team and organizational system where humans and AI work together over complex sources without redefining access control.

LLM role Compose typed views from reachable sources

The core insight

A living doc is not mainly a hand-authored narrative. It is a typed synthesis surface. The type says what kinds of things should converge. The runtime says which sources are actually reachable in this session. The LLM composes the view from that reachable surface.

types define structure sources define access LLM performs composition rendered HTML is a snapshot

What must stay separate

Type registry

Defines convergence types, entity types, status sets, and structural contracts.

Living-doc catalog

Tracks which living docs exist, where they live, who owns them, and how they are synced.

Source access

Stays in GitHub, Figma, AWS, filesystems, issue trackers, internal systems, and SSO/RBAC.

Runtime inference

Uses the current session’s real source surface to populate or update typed sections.

Important correction

The living-doc system should not redefine permissions. It should hook into existing source access systems and inherit what the current session can actually see.

The runtime model

In the current workflow, a session asks for a living doc, the relevant skill is loaded, and the LLM checks which sources it can access now. The quality and completeness of the resulting living doc are therefore functions of runtime capability, not a separate permission model inside the living-doc system.

1. Declared scope

Goal, project, initiative, lab surface, incident, or research track.

→

2. Type language

The convergence registry defines which semantic shapes are available.

3. Runtime capability surface

The session resolves what GitHub repos, Figma files, logs, AWS accounts, datasets, notebooks, or APIs it can actually reach.

→

4. Composition

The LLM gathers entities from reachable sources and fills typed sections with explicit evidence and gaps.

Why this matters for teams

Once humans, teams, org units, and AI are all doing deep work together, source management becomes a complexity surface of its own. A living-doc system only helps if it compresses that complexity rather than reproducing it.

  • Rendered docs are useful snapshots, but not the authority layer.
  • The catalog is the authority for which living docs exist.
  • The type registry is the authority for what kinds of typed surfaces can exist.
  • Runtime access determines what an agent can populate in a given session.
  • Missing access should surface as incomplete coverage, not be hidden behind false completeness.

What the system should and should not do

Keep the role boundaries explicit so the system does not become a second governance layer.

Concern Should do Should not do
Type system Define semantic convergence contracts Carry environment-specific access rules
Catalog Track actual living-doc instances and ownership Redefine source permissions
Runtime Resolve what sources are reachable now Pretend coverage exists where access is missing
Rendered artifact Provide portable, self-contained snapshots Become the canonical authority

The missing layer: a pre-scan skill

Before a living doc is created or extended, a pre-scan skill should inspect the available source surface in light of organizational goals, objectives, and active work. It should discover which convergences are worth turning into living-doc sections or entirely new docs.

Inputs Org goals, project context, active work, current source access, existing type registry, current living-doc catalog
Outputs Proposed scopes, candidate convergence types, source coverage, missing connectors, extend-vs-create recommendation
  1. Inspect current goals and objectives.
  2. Scan the runtime capability surface.
  3. Detect meaningful source combinations.
  4. Match them to existing convergence types.
  5. Identify where a new type may be emerging.
  6. Recommend a new living doc, an extension to an existing one, or no doc at all.

Lab example

Imagine a research manager using a local model over experiment logs, hardware traces, researcher artifacts, notebooks, code repos, and issue trackers. The model does not need a new permission system. It needs runtime access to those sources through whatever the lab already uses.

Possible living-doc surfaces

  • Experiment integrity surface — run logs, anomalies, reruns, notes
  • Hardware readiness surface — calibration, incidents, maintenance, constraints
  • Reproducibility surface — code version, dataset, config, outputs, validation state
  • Result-evidence surface — plots, interpretations, decision records, open doubts

Runtime coverage matters

Two agents may create the same conceptual living doc with different completeness because their accessible source surfaces differ. That is not a flaw. It just has to be explicit. 

requested surface: reproducibility
expected sources:
  - run logs
  - code repo
  - dataset snapshot
  - analysis notebook

runtime coverage:
  - run logs: yes
  - code repo: yes
  - dataset snapshot: partial
  - analysis notebook: no access

What this means for a team version

A team version of the system should not be understood as a shared folder of HTML files. It should be understood as a shared typed reasoning substrate built from three cooperating layers.

Semantic layer Convergence registry stays small, explicit, and inference-strong.
Instance layer Catalog tracks actual living docs, scope, ownership, and freshness.
Runtime layer Skills and agents compose from the actual source capability surface of the current session.
