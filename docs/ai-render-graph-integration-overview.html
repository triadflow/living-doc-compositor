Living Doc Compositor / Issue 135 Stance

AI Render Graph should land as a type-aware, opt-in advisory export layer.

The integration fits this repo if it stays read-only, preserves the normal standalone export path, and hangs its semantics off the existing convergence type system instead of inventing a second AI schema. The exported page may hydrate local inference slots, but it must not become the canonical mutation path.

Opt-in export only Read-only advisory slots Type-aware profiles No Cmd-K replacement Prefer embedded runtime over vendor link
Decision Boundary

What this is for

Use AI Render Graph to add local, runtime-only reading aids to an exported doc: a brief, a staleness note, a risk callout, or a checklist computed from selected sections.

Good fit for exported-doc reader assist
  • Summarize a large section set into a short brief.
  • Highlight which section looks stale, blocked, or under-specified.
  • Generate a local review checklist before sharing.
Hard Boundary

What this is not for

The integration should not alter the meaning of a living doc export. The doc stays readable without a model server, and no runtime hydration path gets to mutate canonical JSON.

Do not turn hydration into authoring
  • Do not replace the existing Cmd-K propose/apply path.
  • Do not let runtime inference write source JSON.
  • Do not require remote hosting or cloud API keys.
Separation Of Concerns

Keep Cmd-K and AI Render Graph sharply distinct.

There are already two different product surfaces in play. One is a local mutation workflow. The other is a runtime advisory layer inside an export. They should share vocabulary, but not responsibilities.

Existing Cmd-K AI pass

  • Runs as a propose/apply workflow.
  • Produces accepted patch items and disk changes.
  • Belongs to authoring and maintenance.
  • Should remain the only AI-assisted mutation path.

Proposed AI Render Graph layer

  • Runs inside an exported document at read time.
  • Fills advisory slots only.
  • Belongs to interpretation, review, and orientation.
  • Must remain optional and non-canonical.
Recommended Architecture

The integration boundary should live across three layers, not one.

The current convergence type system already defines what a section means. AI support should extend that semantic layer, while leaving section enablement and runtime wiring in the document/export surfaces.

Config Split

Put the right information in the right place.

The useful split is semantic defaults at the type layer, enablement at the section layer, and machine-specific runtime config at export time.

Convergence type

Registry declares typed AI profiles.

This answers: "What advisory AI views are valid for this kind of section?" It should not answer where the model runs or which endpoint to call.

  • Allowed profile ids such as brief or staleness-note.
  • Expected output fields and slot shape.
  • Prompt seeds derived from the type's existing semantic contract.
Section or document

Content opts into specific profiles.

This answers: "Do we want advisory hydration here?" It should be possible to enable it for one section, skip it for another, and keep the rest of the document unchanged.

  • Enable only the profiles that help this section.
  • Keep the doc readable without hydration.
  • Do not store generated advisory text as canonical fields by default.
Export mode

Runtime config stays outside the type system.

This answers: "How should this HTML run on this machine?" That includes endpoint, model, file-open mode, and whether the runtime JS is embedded into the export.

  • enabled, endpoint, model, timeoutMs
  • Whether the runtime is embedded or loaded from a local sidecar
  • Validation of the emitted AI Render Graph spec before write
Proposed Export Flow

Render first, enhance second, hydrate last.

The renderer should preserve the normal static export contract, then layer AI affordances on top only when enhancement is requested.

01

Read canonical JSON

Interpret the living doc with the current registry and section semantics exactly as the normal exporter already does.

02

Resolve allowed profiles

Collect AI profiles from convergence types, then filter to the ones explicitly enabled on the selected sections or document.

03

Emit markers and slots

Add data-ai-source to source regions and data-ai-task to read-only output slots without altering core content.

04

Embed validated spec

Generate an AI Render Graph spec from the type-aware profile set and validate it before writing the final HTML.

05

Hydrate locally if available

When the page is opened, the embedded runtime may call a local model server and fill advisory slots. Without that server, the export still works.

Registry Shape

Example: extend convergence types with typed AI profiles.

This keeps the AI affordance anchored to the type system instead of inventing a separate parallel schema.

{
  "convergenceTypes": {
    "verification-surface": {
      "aiProfiles": {
        "brief": {
          "allowed": true,
          "sourceMode": "section",
          "uses": [
            "promptGuidance",
            "sources",
            "statusFields",
            "textFields",
            "detailsFields"
          ],
          "outputs": {
            "headline": "string",
            "summary": "string",
            "nextActions": "string[]"
          }
        },
        "staleness-note": {
          "allowed": true,
          "sourceMode": "section",
          "outputs": {
            "staleArea": "string",
            "reason": "string",
            "suggestedCheck": "string"
          }
        }
      }
    }
  }
}
Section And Export Shape

Enable per section, configure per export.

The document selects which profiles to expose. The export supplies machine-specific runtime details.

{
  "id": "verification-main",
  "convergenceType": "verification-surface",
  "ai": {
    "enabledProfiles": [
      "brief",
      "staleness-note"
    ]
  }
}

{
  "aiEnhancement": {
    "enabled": true,
    "runtime": "file-open",
    "embedRuntime": true,
    "model": "gemma-4-e4b-it",
    "endpoint": "http://127.0.0.1:1234/v1/chat/completions"
  }
}
HTML Contract

The export should prefer embedded runtime, not an external vendor link.

The cleanest version for this repo is to use the library as a build dependency, validate and generate at render time, then inline the runtime into the final HTML so the export remains self-contained.

<section data-ai-source="verification-main">
  ... rendered living doc content ...
</section>

<aside data-ai-task="brief.summary">
  Static fallback summary shown when no local model is available.
</aside>

<script type="application/ai-render-graph+json">
{ "...validated type-aware spec..." }
</script>

<script>
  /* embedded ai-render-graph runtime */
  /* embedded mount code with local endpoint config */
</script>
A separate vendor/ai-render-graph.file.js sidecar is possible, but weaker than the repo's standalone-export contract.
Use Cases

Adopt it first where advisory text is clearly useful and clearly non-canonical.

Good first use cases

  • Convergence brief for a large exported doc.
  • Staleness or risk note across selected sections.
  • Short review checklist before sharing.
  • Reader orientation block for dense verification or proof docs.

Not good first use cases

  • Editing or patching canonical JSON.
  • Replacing section content with model output.
  • Default behavior on every export.
  • Public pages that must behave identically on all machines.

Why the type system matters

  • Different convergence types need different advisory shapes.
  • The registry already describes what each section means.
  • Prompt seeds should come from those semantics, not freeform prompts alone.

Why the export boundary matters

  • Model and endpoint are machine-local concerns.
  • Normal static exports must remain unchanged.
  • The HTML still needs graceful degradation without runtime hydration.
Recommended Scope

If issue 135 moves forward, the narrow version is the right version.

Proceed with an opt-in enhanced export path that generates read-only AI Render Graph slots from registry-aware profiles. Do not turn it into a general AI templating surface, and do not let it collapse the existing distinction between export-time advice and authoring-time mutation.

Short version: integrate the library, but keep the export standalone, keep the hydration advisory, and let the convergence type system define what kinds of AI views are legitimate for each section.
