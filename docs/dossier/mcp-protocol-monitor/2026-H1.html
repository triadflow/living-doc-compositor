MCP Protocol Monitor · 2026-H1

A protocol many builders converge on

Eighteen months in, the Model Context Protocol has become the default connective tissue between AI agents and the systems they need to reach — and it got there without a vendor winning.

Monitoring snapshot · 2026-H1 Source: AI Engineer talk, David Soria Parra, April 2026

The question most technology writing still asks is which company wins. The Model Context Protocol is a quieter and more interesting story: an open specification that went from a few pages of draft text to one hundred and ten million monthly downloads in roughly eighteen months, now governed by a foundation where Anthropic, Google, Cloudflare, OpenAI and a long tail of independent contributors all push the same artifact forward. React, the reference point for software standards absorbing quickly, took about twice as long to reach the same download volume. No one is winning MCP. It is converging.

For anyone whose work sits at the join between knowledge systems, automation and human judgement — and that is an increasingly large room — the shape of this convergence is worth paying attention to, separately from whether AI agents ever do the things their promoters claim. It is rare for an industry to agree on connective tissue without a dominant vendor setting the terms. When it happens, it tends to compound.

What the baseline looks like

As of April 2026, the protocol sits in a characteristic middle state. Its foundational primitives — tools, resources, prompts — are stable and universally supported across major clients. A second tier of primitives, added in the last year, is spec-present but unevenly adopted: elicitation (servers asking the user for input mid-flow), tasks (long-running and agent-to-agent work), and MCP applications (servers shipping their own interfaces, rendered by the client). The last of those was demonstrated in production during the April talk and is already powering at least one Anthropic product feature; it was shipped as an extension rather than core, acknowledging that a command-line client cannot reasonably render HTML.

A third tier — skills over MCP, stateless transport, cross-app access, server discovery — is proposed and targeted for a coordinated June 2026 spec drop. These four additions tell you what the protocol's designers believe is missing. Skills-over-MCP lets a server ship domain knowledge alongside tools, so authors can update how their system should be used without waiting on plug-in registries. Stateless transport, a Google-led proposal, would let MCP servers be deployed the way any other stateless REST service is deployed. Cross-app access folds enterprise single sign-on into the protocol so a user authenticates once with their identity provider and gets onward access to every MCP server their company runs. Server discovery defines a well-known URL convention so crawlers, browsers and agents can find MCP servers the same way they find robots.txt.

The through-line across all four: move what was previously platform-specific plumbing into protocol-level agreement. That is the shape of an ecosystem that expects to be deployed widely, not one that is still looking for its first believers.

The sharpest technical opinion

Stop taking REST APIs and putting them one-to-one into an MCP server. Every time I see someone building another REST-to-MCP conversion tool, it is a bit cringe — because it just results in horrible things. Design for an agent. — David Soria Parra, Anthropic · April 2026

This is the kind of opinion a community holds when it is confident enough to publish taste. The practical point underneath it is the shift from agents orchestrating chains of tool calls to agents writing code against an execution environment — what Anthropic and Cloudflare both call progressive discovery and programmatic tool calling, and what others call "code mode." Give the model a small execution sandbox and structured output types. Let it compose the pieces itself. The measured effect in Claude Code, where progressive discovery replaced upfront tool loading, was described as a massive reduction in context use. Every other client with a large tool inventory has the same problem to solve.

Why the governance shape matters

The part of this story that does not fit the usual arc is the foundation. MCP's stewardship has moved out of Anthropic's hands and into a multi-party body run in the open — Discord, public issues, specification proposals from any member. One of the first significant proposals under the new governance comes from Google, not Anthropic. That is the right sign. A protocol that is actually neutral tends to show it by accepting load-bearing changes from parties other than its original author.

Whether this produces a healthy standards process or merely a well-lit committee is a question for 2027. But the conditions are in place. One hundred and ten million downloads a month creates its own gravity; a neutral governance body removes the strategic incentive for any single vendor to fork; and the proposed June 2026 additions commit multiple vendors to the same surface at the same time. The interesting question is no longer whether MCP persists. It is which parts of work that currently live inside proprietary platforms — CRMs, design systems, knowledge bases, ERPs — become protocol-addressable next.

The connectivity stack, plural

One theme carried throughout the April talk is worth naming explicitly for anyone thinking about how to build on any of this: connectivity is never one thing. The production agents of 2026 will use skills for packaged domain knowledge, command-line tools where a sandbox is already present and the tools were in the model's pre-training, MCP where rich semantics, long-running work, or authorization are needed, and direct computer use where nothing else is available. Anyone who tells you there is a single right answer is selling the one they happen to have built.

What to watch in 2026-H2

  1. The June 2026 spec drop — stateless transport, server discovery, and skills-over-MCP landing together would be the most substantive protocol update since remote servers were added.
  2. Client conformance. The current monitoring baseline stubs every major client as "not-assessed"; by the next check-in there should be a dated profile of what Claude Code, ChatGPT, VS Code, Cursor and others actually implement.
  3. The foundation's body. The April talk said "foundation" without disambiguating between Linux Foundation stewardship and a standalone MCP Foundation. This monitor will name it correctly next period.
  4. Python and TypeScript SDK v2 cuts — committed publicly, absorbing lessons from the community-built FastMCP. The quality of those releases will determine how much of the ecosystem stays on the original SDKs versus migrating.
  5. Whether the stated distaste for one-to-one REST-to-MCP adapters translates into an observable drop in such servers — or stays rhetorical while the tide of wrappers continues.

One artifact. Many contributors. Converging. It is a pattern the software industry produces rarely and rewards disproportionately when it sticks.
