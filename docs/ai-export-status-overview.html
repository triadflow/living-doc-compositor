What Works Now

The important part is that this is no longer only a design direction. There is a coherent implementation path in the renderer and a working exported sample.

Type-aware advisory export exists

The renderer emits advisory regions, a real AI Render Graph spec, advisory metadata, and a local-model runtime path using convergence-type aiProfiles instead of ad hoc per-document prompts.

Standalone export contract is preserved

The exported living doc still behaves as a normal standalone HTML artifact when no model is present. AI is an optional enhancement layer, not a precondition for reading the document.

Local model behavior is materially hardened

  • Near-miss JSON shapes are normalized into the expected typed fields.
  • Malformed JSON gets retry handling before surfacing an error.
  • Nested wrapper outputs are recovered when they still clearly contain the intended profile payload.

Lifecycle persistence is fixed

Successful results now survive later section runs. The exported advisory surface no longer clears earlier successful cards just because the user ran a different section afterward.

Current Boundaries

In

Document-grounded prompts, local-model inference, first-three-type support, standalone HTML, typed advisory slots, retry/error states, and persistence across reruns.

Not Yet

Broader type rollout, tool-backed search, internet-grounded inference, final visual polish, and broader product-default decisions about which profiles should always be shown.

Out of Scope

Canonical document mutation, replacing Cmd-K, turning advisory output into source truth, or making network/tool access the default inference assumption.

Most important structural fact

This is a read-only reader aid attached to rendered exports. The source JSON remains canonical.

Most important remaining product question

How far the first three profile sets should go before widening type coverage. The next leverage is product quality, not more plumbing.

What Changed Recently

Phase 1

Phase 2

Library-Native Runtime

The export switched from a placeholder idea to using the local ai-render-graph runtime and a real typed spec shape.

Phase 3

Recovery Hardening

Prompt handling, alias recovery, malformed JSON retry, and error-state handling were tightened against real local-model behavior.

Phase 4

Persistence Fix

The runtime lifecycle bug was closed so the advisory surface can now act like a stable multi-card reader layer across multiple runs.

What This Means Practically

If someone asks whether this is “real” yet, the answer is yes, but with a specific meaning.

It is real enough to demonstrate in a living doc

You can open an exported living doc, run advisory slots against a local model, and see typed results land back into the rendered page with meaningful state handling.

It is not yet the final product surface

The system is strong on architecture and implementation truth now. The main unfinished work is product maturity: deciding default profiles, shaping stronger copy/visual rhythm, and extending only after the first three feel solid.

The repo is no longer blocked on unknown integration risk

The major uncertainties about runtime embedding, task typing, local-model calls, standalone export behavior, and cross-run stability have already been turned into working code.

Immediate Next Moves

Refine the first three types

Keep tuning prompts, slot language, and per-profile payload shaping using the now-stable surface.

Decide product defaults

Choose which advisory profiles should be visible by default and which should remain explicit opt-ins.

Only then widen coverage

Do not broaden to more convergence types until the current first slice feels materially solid as a reader experience.
