Proposal · ticket #75

Five convergence types for composing a living doc from an OSS issue deep-dive.

The trial doc for Textual #5380 stretched three existing types across five distinct semantic shapes. This proposal names those shapes and gives each its own contract — so the substrate stays reliable as the OSS-triage skill scales.

Every new type defended here preserves a distinct semantic contract that cannot be cleanly carried by an existing one. Volume is not the cost — stretching is.

5 proposed types Trial doc: Textual #5380 Not yet in registry
The lens

Semantic stability, not volume.

Every convergence type is a stable contract: readers — humans and agents — can trust what the type means, what shape of content it holds, what question it answers.

Why stretching is the real cost

When an existing type is stretched to cover an adjacent-but-distinct shape — conflating findings + attempts + related-issues under investigation-findings, for example — the contract erodes. A card under that type could now be three different things, and the reader has to figure out which.

The Living Doc Compositor's whole premise is that types are trustable semantic units. Fewer overloaded types erode that. More types with crisp meaning preserve it.

Registry entries cost something — they do — but that cost is secondary to preserving the semantic contract. If a new type answers a genuinely different question, it earns its place.

Proposed types

Five types. Five distinct questions.

Each is scoped so tightly that it cannot be cleanly collapsed into another existing type without losing meaning.

1attempt-log 2issue-orbit 3code-anchor 4symptom-observation 5maintainer-stance
1
attempt-log
What was tried to resolve this issue, what did it prove, where does it live?
Contract: Each card is one action taken against the problem and its result. An attempt is not an observation — it has a doer, an outcome, and optionally a shipping location. Findings sit under symptom-observation; decisions sit under maintainer-stance. This type is exclusively for tried things.

Card fields

  • id / name — short handle
  • what_tried — prose description of the action
  • what_proved — the outcome and what it revealed
  • shipped_in — commit or PR URL, or null
  • cost_to_apply — rough estimate (LOC, invasiveness)
  • notes[] — supporting links

Status set

probe workaround-shipped rejected superseded

Projection

Card grid (2 columns). Each card prominently shows shipped_in as a link and cost_to_apply as a tag.
// In the Textual #5380 trial { "id": "inspect-ai-patch", "name": "Override _watch_app_focus, re-call _enable_mouse_support on focus gain", "status": "workaround-shipped", "what_tried": "On focus=True, call driver._enable_mouse_support() wrapped in try/except", "what_proved": "Mouse survives VS Code reload in production inspect_ai; ANSI codes are idempotent on xterm.js", "shipped_in": "https://github.com/UKGovernmentBEIS/inspect_ai/commit/8ce3409", "cost_to_apply": "~20 LOC, App-level override, no invasive refactor" }
Not to be confused with: investigation-findings (observed facts — no doer, no shipping site) or decision-record (settled direction — no outcome yet, only rationale).
In the Textual #5380 trial: currently stretched as investigation-findings → should be attempt-log. Affects: suspend-probe, direct-enable-call, inspect-ai-patch.
2
issue-orbit
Which other issues and PRs in this repo share root cause, symmetry, or adjacency with the focal issue?
Contract: Each card is another issue or PR and its relationship to the focal one. This is a graph of siblings, not a list of facts about the focal issue. The card's main payload is the relationship itself.

Card fields

  • id / name — #1171 — Symmetric leak…
  • url — GitHub URL
  • github_state — open / closed
  • relationship — how it relates to focal
  • relevance — prose: why this matters
  • closed_by_pr — optional

Relationship enum

same-root-cause adjacent symmetric superseded prior-art

Status set

closed-fixed closed-wontfix open-active open-stale
// In the Textual #5380 trial { "id": "issue-1171", "name": "#1171 — Coordinates leak to shell prompt after force-stop", "url": "https://github.com/Textualize/textual/issues/1171", "github_state": "closed", "status": "closed-fixed", "relationship": "symmetric", "relevance": "Same state-drift family as #5380: disable_mouse_support never runs on force-kill, so xterm.js keeps emitting coords. Shows the bidirectional sync gap." }
Not to be confused with: investigation-findings. A finding is an observation about the focal issue. An orbit card is metadata about a different issue and how it relates. Collapsing them loses the graph structure — and the ability for a reader to see "what other bugs in this family have already been walked."
In the Textual #5380 trial: currently stretched as investigation-findings → should be issue-orbit. Affects the entire related section: issue-5547, issue-1171, issue-1031, issue-5894.
3
code-anchor
Which code locations matter for this issue, pinned to a specific revision?
Contract: Each card is one exact file-and-range, pinned to a commit SHA or tag, with a short description of what the code does and why it's load-bearing for the focal issue. It is a diagnostic pointer, not a product capability.

Card fields

  • id / name
  • path — repo-relative
  • range — line or line range
  • revision — commit SHA or tag
  • what_it_does — one-line code-level summary
  • why_it_matters — relevance to focal issue

Status set

current changed-since-issue deprecated

Freshness signal

When the pinned revision is older than main, the renderer should flag whether the anchor's file/range has changed, so the reader knows if the pointer is stale.
// In the Textual #5380 trial { "id": "enable-mouse-support-once", "name": "_enable_mouse_support() — single call site", "path": "src/textual/drivers/linux_driver.py", "range": "248", "revision": "f8b42bd7a93578338f1b3668a1dc83f79e344fe8", "status": "current", "what_it_does": "Only call to _enable_mouse_support() inside start_application_mode; never re-runs", "why_it_matters": "This is the synchronization gap — xterm.js reloads have no signal path back here" }
Not to be confused with: capability-surface. A capability is a product-framing — "the app supports mouse input." A code-anchor is diagnostic — "line 248 of linux_driver.py fires the enable once." One surfaces user-facing behavior; the other points into source for an engineer.
In the Textual #5380 trial: currently stretched as capability-surface → should be code-anchor. Affects: enable-mouse-support-once, ansi-codes-written, disable-mouse-support-mirror, watch-app-focus-hook.
4
symptom-observation
What behavior was observed, how do you reproduce it, who witnessed it?
Contract: Each card is one observed behavior with a reproduction path and a witness. The presence of repro steps and witness attribution is the semantic contract — without them, it's just a generic finding.

Card fields

  • id / name
  • environment — OS, terminal, versions
  • repro_steps — how to reproduce
  • witness — who + when + link
  • contradicted_by — optional counter-observer

Status set

reproduced unconfirmed contradicted

Why not just findings?

The repro path and witness are load-bearing — they're what lets a subsequent reader verify or contest the observation. Generic findings don't require them; symptoms do.
// In the Textual #5380 trial { "id": "reload-drops-mouse", "name": "VS Code reload drops mouse tracking silently", "status": "reproduced", "environment": "Ubuntu 22.04, VS Code 1.95.3, Textual 0.89.1, xterm-256color", "repro_steps": "1. Run stopwatch.py from textual/docs/examples/tutorial 2. Trigger 'Developer: Reload Window' 3. Mouse events stop; keyboard still works", "witness": "jjallaire-aisi, 2024-12-11 (issue body)" }
Not to be confused with: investigation-findings. A symptom-observation is specifically a reproducible behavior in the problem domain, with a provenance trail. A generic finding can be a conclusion, a regularity, or a meta-observation. Keep the symptom layer pure so the "can I reproduce this?" question has one place to look.
In the Textual #5380 trial: currently stretched as investigation-findings → should be symptom-observation for the user-facing behavioral cards (reload-drops-mouse, tmux-immunity, suspend-resume-restores). The idempotency-asserted card is genuinely a finding — stays where it is.
5
maintainer-stance
What position has a named stakeholder expressed, when, and how has it evolved?
Contract: Each card is one named stakeholder's position on a question, with the rationale they gave and any rebuttal in the thread. Stances are not settled — they're positions in an evolving conversation that may never close. This is what separates them from decision-record.

Card fields

  • id / name
  • stakeholder — handle + role
  • stated_at — ISO + comment URL
  • position — the stance itself
  • rationale — reasoning given
  • rebuttal — optional pushback card
  • evolution — has it shifted since?

Status set

current softened retracted unchallenged

Social shape

Stances have owners and timestamps. Decisions typically don't — they just are. If the next contributor has to re-litigate a stance, the doc needs to preserve the original debate, not paper over it.
// In the Textual #5380 trial { "id": "maintainer-stance", "name": "Will McGugan — 'not ours to fix'", "status": "current", "stakeholder": "willmcgugan (Textual maintainer)", "stated_at": "2024-12-11T… (comment URL)", "position": "This is a VS Code bug, not a Textual bug; the reporter should raise it upstream.", "rationale": "xterm.js is the party resetting mouse state on reload; Textual is acting correctly.", "rebuttal": "jjallaire-aisi: VS Code bugs of this type don't get prioritized; Textual has stronger incentive and a small, idempotent fix.", "evolution": "No follow-up from maintainer after rebuttal (as of 2026-04-18)." }
Not to be confused with: decision-record. A decision is something the team chose — it has weight because it's settled. A maintainer stance is a current position that may be wrong, out of date, or contested. Collapsing stances into decisions loses the social dimension the next contributor needs to work the issue.
In the Textual #5380 trial: currently stretched as decision-record → should be maintainer-stance. Affects: maintainer-stance, user-pushback, no-pr-opened, open-question-gating.
