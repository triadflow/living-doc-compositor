1. The inference-time re-derivation tax

Today, when an agent opens a living doc it reads the objective, scans the sections, and silently re-derives a set of things that the document never states outright:

  • Why each section uses the convergence type it uses.
  • Which parts of the objective each section carries.
  • Which cross-section contracts hold (e.g. "decisions don't live inside verification").
  • Whether the current set of sections actually covers the objective.

The registry at scripts/living-doc-registry.json gives every convergence type a structuralContract, a notFor list, and promptGuidance. That is a generic vocabulary — useful, but the document-local justification (why this type here) is recomputed on every pass, by every agent. That recomputation is the tax.

The cost

Two Codex workers looking at the same doc re-derive the section taxonomy slightly differently. Coverage of the objective is assessed by re-reading prose. Invariants are enforced only when a reviewer notices. Every action on the doc pays the tax again.

2. What meta-layer data captures

Four things that are implicit today become explicit:

Objective decomposition

The free-form objective string is split into typed facets. Each facet has an id, name, and description. "UI fidelity", "repo-connect", "push-runtime", etc. become addressable units.

Section rationale

Each section carries a rationale field alongside its convergenceType: the document-local justification for why this shape was chosen, once, in writing.

Coverage edges

A list of { facetId, sectionId, cardId } tuples that map each facet to its carriers. Coverage becomes a set-difference operation, not an LLM re-read.

Local invariants

Doc-specific contracts sharper than the registry's generic notFor. Example: "every Capability Map card must carry at least one codePaths entry".

3. Why not a sibling convergence type

Making meta-data just another convergence type — a section called "Meta" sitting next to Capability Map and Verification Plan — is tempting for symmetry, but it conflates data that governs interpretation with data that is interpreted. Three concrete tells:

TellWhy it matters
Load order An agent needs meta before sections, to know which section carries what. Peer sections have no structural precedence — nothing in the schema says "read me first."
Derivation direction Every existing convergence type derives state from entity sources (code, tickets, figma). Meta derives state from other sections. Circular at the projection layer, clean at the doc-root layer.
notFor shape Every type declares what it is not for. Meta isn't "not for X" — it is about the whole doc. The contract shape doesn't fit.

Analogy: the registry itself is not a convergence type in a doc — it governs docs from outside. Meta-data is a doc-local extension of the same kind of governance.

4. The two-layer split

Meta-data lives at the doc root, as a peer of title, objective, and sections. A new convergence type, coherence-map, renders a projection of that meta as a visible section. The data is authored once at the root; the section is a mirror.

Document shape

Tier 1 · Doc root (governance)
docId title objective objectiveFacets coverage invariants sections
↓ renders as ↓
Tier 2 · Sections (peers)
coherence-map status-snapshot capability-surface operating-surface tooling-surface verification-surface decision-record

The distinction

The data is doc-root. The view is a convergence type. The registry's job is to vocabulary-ify section shapes; adding coherence-map extends that vocabulary in the direction it already points. Adding objectiveFacets / coverage / invariants extends the schema, not the vocabulary.

Two fields on the type are new and central to the proposal: derived: true and derivedFrom: [...]. They signal to the renderer and to agents that data[] is computed from doc-root fields, not authored. That's the contract that keeps the view a mirror.

8. Inference-time gains

Each new field collapses a piece of LLM re-derivation into either a deterministic check or a cached read.

Worker question Before After
Does this doc still cover the objective? LLM re-reads prose, guesses. Set-difference over coverage edges. Non-LLM.
Which section should this new fact go in? LLM decides by shape-matching. coverage points directly at the right card.
Is my proposed edit legal? Reviewer catches contract breaks. Invariants checked at plan time.
Why is this section this shape? LLM re-derives structural contract. section.rationale carries the answer.
Is this section stale? Timestamp heuristic. Semantic triggers via dependsOn.
Where do I start? Scan all sections. Load meta, score coverage, jump.

Compounding effect

The meta layer is roughly 10% of the document body. Agents can load it, reason about where to work, and only expand the relevant card grid. Multi-pass reasoning stays cheap. Multi-agent coherence is structural rather than stochastic: two workers read the same rationale, enforce the same invariants, and score the same coverage.

9. Worked example — mobile workstream

Take the existing mobile companion workstream doc. Its objective names eight facets. Its six sections (status snapshot, capability map, runtime state model, operational tools, verification plan, open decisions) each carry a subset of those facets.

Before — implicit

An agent opening the doc must:

  1. Read the eight-part objective sentence and internally list the facets.
  2. Re-read each section and decide which facets it covers.
  3. Notice that pixel-match fidelity is mentioned in both Capability Map and Verification Plan, and conclude it is "under-covered" (gate still open) by reading prose.
  4. Hope that the next Codex worker reaches the same conclusions.

After — explicit

Same agent opens the doc and:

  1. Loads objectiveFacets (8 items) and coverage (11 edges) from the root.
  2. Computes coverage in one pass: every facet has ≥1 carrier, eas-setup is covered by one section only, ui-fidelity is flagged under-covered because its verification gate is still open.
  3. Sees rationale on each section explaining the structural choice — no re-derivation.
  4. Validates any proposed edit against the three invariants before writing.

The shift

The document stops being only the output of a thought process and starts carrying that process inline, where every downstream agent can read, verify, and extend it without re-deriving it. Today the registry is the grammar and the doc is the sentence. Meta-layer data adds the reason this sentence was the right one to write — so future edits reason against intent, not shape alone.
