Internal Overview: Living Doc Lifecycle Harness

A source-grounded map of the standalone lifecycle harness as a control system around model runs. This page is internal system documentation, not a magazine article.

Created 2026-05-12 Evidence: code, tests, living docs, run artifacts, dashboard Out of scope: magazine glossary as system evidence

System Claim

The lifecycle harness is the controller-owned loop that runs model work against a living doc, finalizes each iteration from artifacts, selects the next unit, and permits terminal closure only through objective proof or explicit user stop.

Evidence Boundary

The magazine glossary may help writers find prior vocabulary, but it is not evidence for runtime behavior. Runtime claims on this page must ground in harness scripts, contract tests, living-doc JSON, run artifacts, or dashboard graph behavior.

scripts/living-doc-harness-iteration.mjs Iteration finalizer. Builds evidence, reviewer verdicts, post-review selections, side-effect gates, closure review, terminal state, proof and dashboard artifacts. How one worker result becomes a routed next unit or terminal state.

Unit Registry

The registry is the controller's unit vocabulary. It declares what each unit accepts, what evidence it must inspect, what verdicts it may emit, and which next unit types it may request.

Core Rule

Worker output is evidence, not authority. Reviewer output is a routed verdict, not always terminal authority. Closure requires the controller's proof path, and non-closure must become a next contract rather than disappear into chat.

Data That Must Stay Durable

  • Living doc path, objective, success condition, current state.
  • Required inspection paths and run configuration.
  • Worker evidence, native trace references, changed files.
  • Reviewer verdict, reason codes, proof gates.
  • Terminal record, output-input handoff, next selected unit.

Dashboard Responsibilities

  • Build graph nodes with graphNode.
  • Build handoff edges with graphEdge.
  • Expose active inference unit state.
  • Render node/edge inspector details.
  • Expose commitIntent on repair nodes and changed-doc edges.

Non-Goals

  • Do not treat a blog glossary as runtime evidence.
  • Do not let worker confidence persist closure alone.
  • Do not treat blocker issue creation as success.
  • Do not let post-flight summary create closure.
  • Do not hide continuation behind a terminal-looking state.

Current Open Questions For The System

Which contracts need schema-level examples?

Several contracts are named in the registry. The overview should eventually link each unit type to a minimal valid input and output artifact.

Which dashboard edges are authoritative?

The graph currently represents lifecycle handoffs, repair chains, living-doc changes, and side-effect evidence. The next documentation pass should enumerate edge ids and their contract payloads from fixtures.

Where should live run artifacts be summarized?

Local .living-doc-runs/ artifacts are not stable public evidence. A sanitized evidence index should say which run examples are safe to reference internally.
