AI actions — current wiring

How the Cmd+K AI-pass palette on a flow-body card resolves which actions to show, as it stands today in the compositor.

One sentence

A card's section has a convergenceType. The palette merges registry.generalAiActions (shown for every card) with registry.convergenceTypes[type].aiActions (shown only for cards whose section is that type). No other linkage exists.

The registry is the source of truth

scripts/living-doc-registry.json owns the action vocabulary. Two tiers:

Runtime flow: Cmd+K on a flow-body card

1 · Keybind
Cmd / Ctrl + K
Global listener fires when a .l2-card-selected card is focused.
2 · Resolve
section → type
Look up section by sectionId in #doc-meta, read its convergenceType.
3 · Merge
general + type-specific
actionsForCard() concats both arrays, tagging each with scope.
4 · Render
palette opens
Two groups — General and Type-specific. Enter runs, Esc closes.

The merge, exactly as it happens

// docs/living-doc-compositor.html
function actionsForCard(sectionId) {
  const reg     = registry();
  const meta    = docMeta();
  const section = (meta?.sections || []).find(s => s.id === sectionId);
  const typeId  = section?.convergenceType;
  const typeDef = reg.convergenceTypes?.[typeId] || {};

  const general  = (reg.generalAiActions || [])
                     .map(a => ({ ...a, scope: 'general' }));
  const specific = (typeDef.aiActions || [])
                     .map(a => ({ ...a, scope: 'type-specific', typeId }));

  return [...general, ...specific];
}

What picks the action up after you hit Enter

  • The palette posts to the local ai-pass server on localhost:4322.
  • Engine choice (claude or codex) comes from GET /api/ai-pass/engines; user preference is written back via PUT /api/ai-pass/engines/preferred.
  • Only the action's id, the card payload, and the section's convergenceType travel to the engine. The name and description are UI-only labels.
  • Result comes back as a proposed diff → rendered in the sidebar → user applies or discards.

Things that are not wired

  • No per-entity-type actions (actions attach to convergence types, not the entity types inside a card).
  • No per-status filtering (e.g. "only show check-monotonic when the rung is ready").
  • No per-card override — a card cannot add or hide an action for itself.
  • No conditional visibility based on available sources (e.g. hide GitHub-refresh if no ticket id).
  • No ordering hint — display order is insertion order from the registry.

Adding an action today

Edit scripts/living-doc-registry.json directly. Either append to generalAiActions or add/extend convergenceTypes.<id>.aiActions. No renderer change, no compositor change — the palette reads it on next load.
