Living Doc Agentic Harness Loop

Codex can already complete a bounded A-Z living-doc run when the objective fits the time window. The next step is to treat every stop as a diagnostic signal, invoke the right repair skill, rebalance the living doc, and resume without making the user the constant control layer.

Core shift

  • Run Codex headless against a living doc objective.
  • Classify stops as inference ambiguity or state imbalance.
  • Use skills to repair the doc and source-state interpretation.
  • Resume until closed, blocked, pivoted, deferred, or budget exhausted.

The Iteration Loop

The harness does not blindly keep prompting. It uses the living doc as the state surface and the skill bundle as the repair mechanism between inference attempts.

01 Load living doc

Read objective, acceptance criteria, convergence types, stage state, proof surfaces, and source-system references.

02 Run Codex headless

Give Codex a bounded interval to work from A to Z against the living doc.

03 Capture stop

Record why the run stopped: question, uncertainty, tool failure, closure claim, loop, or partial state.

04 Classify stage

Infer whether the doc is raw, shaped, executable, in motion, out of balance, or a closure candidate.

05 Invoke skill bundle

Run shaping, readiness, balance, conservation, activation, or path validation based on the stop signal.

06 Patch living doc

Update objective, criteria, proof, issue state, blocker, deferral, pivot, or repair notes.

07 Verify and render

Run proof commands, inspect source-system state, render the doc, and keep the current state portable.

08 Resume or exit

Resume Codex with repaired context, or exit as closed, blocked, pivoted, deferred, or budget exhausted.

Stop Signals

A stop is not automatically a failure. It is evidence that the inference run has reached ambiguity, imbalance, missing proof, or closure pressure.

Clarification question

The objective, scope, source surface, or acceptance criterion is not explicit enough for independent execution.

No clear next action

The living doc may be shaped but not executable; the next valid operation is not visible.

Tool or proof failure

The source system is contradicting the expected path, or the proof surface was underdefined.

Closure claim

The run says done; the harness must challenge whether every objective term is accounted for.

Repeated planning

The model is reorganizing intention instead of changing the source state or proof state.

User-style handoff

The model stops with a summary that expects the user to decide what the living doc itself should decide.

Skill Routing

The harness chooses skills from the stop condition and the living-doc stage. This is the orchestration layer that currently happens manually while waiting for user feedback.

objective-acceptance-shaping Objective is still soft.

Critique rough intent, propose the first clear objective, and attach acceptance criteria that can fail.

objective-execution-readiness Doc may not be executable.

Check whether the objective, criteria, proof surfaces, and next action are enough to drive work.

living-doc-balance-scan State seems unstable.

Classify the imbalance before choosing repair: objective, proof, governance, source, or stage confusion.

equilibrium-rebalance Surfaces disagree.

Repair the doc when source state, issue state, proof, invariants, or user feedback pull apart.

activation-energy-review Closure is claimed.

Test whether proof crossed the objective threshold instead of accumulating closure-shaped artifacts.

catalytic-repair-run Repair must be executed.

Patch the doc, update issues, verify source state, render, and commit the repair before resuming.

Harness Contract

The harness should be strict about what it treats as progress. A nicer summary is not progress unless the living doc state or source-system truth changed.

A stop is a control point. The harness must classify it, repair the living doc if needed, and resume only with a clearer objective state. 
while budget_remains:
  state = read_living_doc()
  result = run_codex_headless(state, timebox)

  stop = classify_stop(result)
  stage = infer_living_doc_stage(state, result)

  if result.claims_closure:
    audit = run_closure_skills(state, result)
  else:
    audit = run_repair_skills(stage, stop, state, result)

  patch = produce_doc_patch(audit)
  apply_patch_and_render(patch)

  if audit.exit_state in ["closed", "blocked", "pivoted", "deferred"]:
    break

  resume_with_repaired_doc()

Exit States

The loop should not run forever. It needs a small set of honest end states that preserve the objective.

closed Objective proved.

Every accountable term is satisfied and the proof surface agrees with the source system.

blocked Rate-limiting condition.

A missing permission, source, decision, dependency, or failing proof prevents further progress.

pivoted Objective changed honestly.

The old objective is not claimed as complete; the new objective and carry-over are explicit.

deferred Outside current objective.

A remaining part is named with a trigger for reopening, not hidden as vague future work.

budget-exhausted Run stops without pretending.

The harness records what remains false and what skill or proof surface should run next.

resume Repaired enough to continue.

The living doc has been patched and the next valid operation is visible again.

Why This Matters

The value is not only less user intervention. The deeper value is making stronger inference runs safer to let continue.

Current manual mode

The model works until it stops. The user notices uncertainty, drift, or premature closure and manually invokes the next skill or correction.

Harness mode

The stop itself triggers diagnosis. Skills repair the living doc, update proof state, and resume the model without making the user carry every control transition.

Next capability step

As models can do longer A-Z runs, the harness gives them durable objective state, stage awareness, and closure gates so capability does not amplify drift.

Proof And Dashboard Layer

The harness needs a second surface around the loop: frozen contracts, state events, sanitized evidence, blocker semantics, and native inference trace links.

Why it matters

A stop should become dashboard-visible proof state, not only a final assistant message. The operator must be able to inspect what stopped, what skill ran, what changed, and whether the run can resume.
