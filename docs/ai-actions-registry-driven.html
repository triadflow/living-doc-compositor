AI actions — registry-driven handlers

A proposal: invert the wiring so adding a new AI action becomes one registry edit, matching how convergence types already work.

Proposal · not yet implemented

Thesis

The handler contract — what to read, what shell commands to run, what patch shape to emit — belongs in living-doc-registry.json, alongside the action's id/name/description. The skill becomes a thin dispatcher over a declarative spec. The registry stays the vocabulary.

Where we are today (for contrast)

Adding an AI action currently has two sides. See current wiring for the full story.

Side 1 · registry

Append one entry. Palette picks it up.

{ "id": "check-still-holding",
  "name": "Check still holding",
  "description": "..." }

Side 2 · two skill files

Edit a handler block into both engine prompts.

## `investigation-findings` → `check-still-holding`

1. Read `card.claim`, `card.evidence[]`...
2. `gh issue view ...`
3. Emit `card-update` when ...
4. Otherwise empty + warning ...

Side 2 drifts. Prompts per engine diverge. The action works or doesn't, depending on which engine is picked. A declared action with no handler silently returns meta.warnings: ["not yet implemented"].

The inversion

Move the handler contract into the registry. Each action carries a machine-readable spec describing its mechanical zone (reads, shell, emit shapes) and a scoped judgment zone (a typed prose slot where the LLM still reasons). The skill becomes a dispatcher that executes the mechanical parts and hands the judgment parts to the model, then assembles the patch.

Now · prose handler per engine

skill-prompt.md (× 2 engines)
  └─ big markdown with
     one handler block per
     action id

registry.json
  └─ id / name / description

Proposed · declarative contract in registry

registry.json
  └─ aiActions[id] = {
       id, name, description,
       scope, appliesTo,
       contract: {
         read, shell, emit,
         judgment, fallback
       }
     }

skill-prompt.md (× 2 engines)
  └─ thin dispatcher
     over contract

The contract shape

The contract has five slots. The first three are mechanical. The last two encode where the LLM is allowed to think.

SlotZonePurpose
read mechanical Declared context — which parts of the card, section, and doc-root the handler needs. Dispatcher loads these before prompting.
shell mechanical A list of safe, templated shell invocations (git, gh) with named results. Templates interpolate only from read output — no free-form strings from the model.
emit mechanical Declarative emit rules — { if: <expr>, kind: <patch-kind>, fields: { ... } }. Kinds and fields are constrained to ai-patch-schema.json.
judgment LLM prose A typed slot where the model writes free-form reasoning — e.g. style-match a neighbour card, or pick among ambiguous commit candidates. The prompt and output shape are fixed; the words inside are not.
fallback mechanical What to emit when evidence is missing — usually empty+warning. Never guess.

Worked example 1 — check-revision-drift (mostly mechanical)

Today: a 30-line prose handler in each of the two skill files. Tomorrow: one JSON entry, zero prose per engine. 

{
  "aiActions": {
    "check-revision-drift": {
      "name": "Check revision drift",
      "description": "...",
      "scope": "type-specific",
      "appliesTo": ["code-anchor"],

      "contract": {
        "read": ["card.revision", "card.path", "card.range"],

        "shell": [
          { "id": "driftLog",
            "run": "git log --oneline ${card.revision}..HEAD -- ${card.path}",
            "if":  "card.revision.nonEmpty" },
          { "id": "latestSha",
            "run": "git log -1 --format=%H -- ${card.path}",
            "if":  "driftLog.nonEmpty" },
          { "id": "headSha",
            "run": "git rev-parse HEAD",
            "if":  "card.revision.isEmpty" }
        ],

        "emit": [
          { "if": "card.revision.isEmpty",
            "kind": "card-update",
            "fields": { "status": "current", "revision": "${headSha}" },
            "rationale": "First revision pin — file exists at HEAD." },

          { "if": "driftLog.isEmpty",
            "kind": "summary-only",
            "summary": "revision still current for ${card.path}" },

          { "if": "driftLog.nonEmpty",
            "kind": "card-update",
            "fields": { "status": "changed-since-issue", "revision": "${latestSha}" },
            "rationale": "${driftLog.lineCount} commits touched ${card.path} since pinned revision." }
        ],

        "fallback": { "onShellError": "empty+warning" }
      }
    }
  }
}

Worked example 2 — enrich-notes (judgment-heavy)

Pure mechanical rules can't write good prose. The contract declares a judgment slot with a fixed prompt and a typed output shape. The dispatcher feeds read output into the slot; the model returns the structured prose; the dispatcher wraps it in a patch. 

"enrich-notes": {
  "scope": "general",
  "contract": {
    "read": ["target-card", "sibling-cards"],
    "shell": [],
    "judgment": {
      "prompt": "Match the shape and voice of sibling cards. Do not invent facts.",
      "output": {
        "type": "array",
        "items": { "role": "callout|reference|paragraph", "tone": "info|warn", "text": "string" }
      },
      "preserve": "existing-notes"
    },
    "emit": [
      { "kind": "card-update",
        "fields": { "notes": "${judgment.output}", "updated": "${now}" },
        "rationale": "Enriched notes to match neighbour style." }
    ]
  }
}

Worked example 3 — decompose (mixed)

Some actions need both zones at once. decompose reads the umbrella card, asks the model to propose sub-units (judgment), then mechanically emits a ticket-create + card-create + coverage-add per sub-unit. 

"decompose": {
  "scope": "general",
  "contract": {
    "read": ["target-card", "sibling-cards", "doc-root.objectiveFacets"],
    "judgment": {
      "prompt": "Propose 3–8 scoped sub-units. Each should be shippable in isolation.",
      "output": {
        "type": "array",
        "items": { "name": "string", "rationale": "string", "carriesFacets": "facetId[]" }
      }
    },
    "emit": [
      { "forEach": "judgment.output",
        "kind": "ticket-create",
        "fields": { "title": "${item.name}", "body": "${item.rationale}" },
        "linkTo": "new-card.ticketIds" },
      { "forEach": "judgment.output",
        "kind": "card-create",
        "fields": { "name": "${item.name}", "status": "planned" } },
      { "forEachPair": ["judgment.output", "item.carriesFacets"],
        "kind": "coverage-add",
        "fields": { "facetId": "${facet}", "cardId": "${card.id}" } }
    ]
  }
}

Runtime flow under the new model

1 · Palette
Cmd+K fires
Same as today — section type → action list.
2 · Server
looks up contract
registry.aiActions[id].contract is the authority.
3 · Mechanical
reads + shell
Dispatcher gathers context and runs declared git/gh commands.
4 · Judgment
LLM slot
Model called only with the scoped prompt + typed output. Engine choice stays user-visible.
5 · Emit
assemble patch
Apply emit rules, run the existing validator, return diff sidebar.

Why this is worth doing

Pros

  • One edit per action. Matches how convergence types are added.
  • Engine parity. Same contract, any engine — no per-prompt drift.
  • Introspection. The "what actions exist and what do they do" page becomes auto-generated.
  • Safer shell. Templates interpolate only from structured read output — no arbitrary model-authored commands.
  • Testable. Contract can be unit-tested with a mock engine; no prompt round-trip needed for mechanical paths.
  • Honest fallbacks. "Not yet implemented" stops being silent — it's a missing contract, caught by a registry lint.

Cons

  • The contract is a DSL. DSLs accrete features; interpolation and conditional expressions need careful scope limits.
  • Real refactor. Two skill files full of prose become one dispatcher each — rewrite, not addition.
  • Judgment-heavy actions may chafe. If the typed output shape is wrong, the contract gets in the way rather than helping.
  • Schema churn. Registry schema version bumps become load-bearing; migrations for existing docs must be handled.
  • Loses the "just write a prompt" escape hatch. Future exotic actions might want freer prose than a contract allows.

Migration path (incremental)

Don't big-bang it. Ship the substrate, migrate one action at a time, let the two styles coexist.

  1. Add the contract substrate. Extend the registry schema; teach the server to call a new dispatchContract(action, ctx) path when action.contract is present. Skills keep handling the rest.
  2. Migrate the easiest action first. check-revision-drift is almost pure mechanical — great pilot. Delete its prose handler from both skills.
  3. Build the judgment-slot runner. Add the typed prompt/output path. Migrate enrich-notes.
  4. Migrate the rest in small PRs. Each migration is one registry edit + two skill deletions + a test.
  5. Auto-generate the wiring page. Once all actions have contracts, the explainer writes itself from the registry.
  6. Retire the prose-handler fallback. When the last action migrates, the skills become pure dispatchers. Lint the registry to require contract on every action.

Open questions

  • Where does the judgment slot sit on the engine-parity spectrum? Same typed output, different models — do we accept quality variance, or strictly-match outputs with a small arbitration layer?
  • How do we express multi-step judgment (e.g. decompose: first propose units, then for each unit write a ticket body) without reintroducing freeform prompts?
  • Are conditional expressions (if: "driftLog.nonEmpty") worth a tiny expression language, or should we restrict to present/empty/equals and force anything richer into a small named helper?
  • Should the contract include retry / refinement when validator rejects the patch, or is that dispatcher concern?
  • What part of today's skill self-check pass becomes a declared contract rule, vs. stays a prose principle?
