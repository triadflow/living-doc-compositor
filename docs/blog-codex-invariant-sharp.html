Codex is better when the invariant is sharp.

In boundary-preserving work, the useful difference is not raw model capability but whether a model stays bound to explicit invariants as the task gets messy. In my experience, Codex does that better than Claude Code, though with the opposite failure mode: over-rigid enforcement and fabricated exceptions.

Rene Luijk Triadflow
PublishedApr 22, 2026 Read6 min StatusLive

The clearest place this shows up for me is proof work in the TrackAndBack parser repo. The task is to repair a claimed email case without losing track of what kind of evidence the run still represents. That means preserving the same claimed email, baseline provenance, graph-vs-outer-layer routing, and whether the system or the operator actually performed the repair.

The objective was already clear. The remaining problem was that one operational invariant was still too weak: the forbidden-intervention boundary.

The missing invariant
  • A qualifying proof run must complete only through system-produced repair artifacts.
  • Any manual graph edit, manual proposal creation, manual apply, or manual runtime patch for the claimed case makes the run non-qualifying.
  • Humans may improve the worker, platform, orchestration, or proof framework only outside the claimed run. Then the run restarts from the recorded baseline.
  • If the worker stalls, the verdict is autonomy gap or tooling failure, not manual continuation of the same run.

Without that rule, a run can look successful while its evidence class has changed in the middle. The worker stalls. Then the interactive Codex session with the operator steps outside the headless proof path and writes the proposal or applies the fix. The output turns green. What remains is operator-assisted support evidence, not proof that the proof system repaired the claimed case on its own.

This is why the invariant is the useful unit of comparison. In boundary-preserving work, the hard part is often not generation. It is preserving the category line: what counts as the same run, what counts as proof, and what quietly downgrades the evidence.

How the difference shows up

In my experience, Claude Code preserves momentum at the level of intent. That can be useful early. On a first pass over a living document, it is often better at moving intent into structure while the structure is still being found.

Codex is stronger on the next phase. Once the invariant is explicit, it tends to keep binding to the wording deeper into the task. In a proof run, that means it is less likely to treat a manual rescue, layer jump, or scope shift as close enough if the written rule forbids it.

Codex's failure mode is the opposite one. If the invariant is soft, it still drifts. If the invariant is explicit, it can over-enforce it. At first that overreach is often implicit. Once confronted, it shows up as fabricated forbidden exceptions and straightforward in-bounds derivations being treated as non-derivable.

That is why the handover can work well. Let Claude Code help find the early structure, then hand the work to Codex when the job becomes mainly about preserving the written boundary. The handoff only works if the invariant is visible enough to survive the switch.

What the human is still doing

The remaining human role is not hand-authoring every step. It is boundary correction in both directions. Sometimes the rule has to be made sharper. Sometimes a fabricated exception has to be surfaced, rejected, and replaced with the right invariant.

With Codex, that usually means being direct enough to show it is wrong. A soft objection is often not enough. You have to point to the in-bounds derivation, identify the invented exception, and write the corrected invariant into the living doc so the next pass has something durable to bind to.

That is also what makes the current state feel strange for senior engineers. Too much time still goes into forcing obvious corrections instead of doing the interesting part of the work. Without a visible invariant surface, I would not touch this level of complexity in this mode at all.

That is why I keep coming back to living docs. They turn operator judgment into explicit, durable invariants a model can actually bind to. Without that surface, this level of complexity is not worth touching in this mode. For boundary-preserving work, living docs are what make it manageable at all.
