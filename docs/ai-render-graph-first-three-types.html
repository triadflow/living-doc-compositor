AI Render Graph / Prompt And Export UX Spec

The first three convergence types should define concrete advisory profiles, not generic AI slots.

This first-pass spec covers three types that exercise different reasoning modes: product-surface convergence, evidence-rich verification, and staged proof. For each type, the export layer should know which advisory profiles are valid, what source material to feed them, what JSON shape to expect back, and where the resulting read-only slots belong in the rendered HTML.

Selection Logic

Why these three

They are not three variants of the same section. Together they cover one shipping surface, one evidence surface, and one staged-proof surface, which is enough to validate the type-driven approach before widening the set.

Prompt Rule

Prompts must be type-seeded

Each prompt should be composed from the convergence type's existing semantic contract: promptGuidance, source fields, status fields, text fields, and details fields. No freeform prompt authorship should be required per document.

UX Rule

Slots must feel like overlays

The export should present advisory results as clearly secondary to the canonical section content. They should read like local review aids, not like new section data.

Delivery Type design-code-spec-flow

Design-Code-Spec Flow should get delivery-oriented advisory views.

This type is about one product surface seen through design intent, code refs, spec refs, interactions, and tickets. The AI layer should help a reader orient to drift and next review moves, not produce generic feature summaries.

surface-brief

Compress the section into one short headline, one summary paragraph, and a short list of the most important current delivery moves.

alignment-risk-note

Identify the most likely design/code/spec drift point and explain why it matters to the shipped surface.

Inline risk callout

review-checklist

Generate a short local checklist for a human reviewer walking the surface before share or handoff.

Collapsible checklist panel

What the renderer should feed

Summarize the surface as a convergence of intent, implementation, and interaction. Call out the most important current alignment risk without collapsing the section into generic implementation status.

Verification Type verification-surface

Verification Surface should produce evidence-oriented advisory views.

This type is already rich in evidence fields such as coverage, next step, gaps, automation, APIs, interactions, and tickets. The AI layer should help compress readiness truth and highlight where evidence is weak or missing.

verification-brief

Summarize the section's readiness posture in plain language while preserving the difference between current coverage and missing evidence.

Top-of-section brief card

weakest-signal-note

Name the probe or block where evidence is currently weakest, and explain whether the weakness is missing automation, unclear API surface, or unresolved gaps.

Inline weakest-signal note

next-verification-loop

Generate a compact next loop: what to check next, what evidence would count, and which gaps should be retired first.

Action block below summary
Prompt ingredients

What the renderer should feed

  • promptGuidance, especially coverage, next step, gaps, automation, and APIs
  • Probe status and priority from each card
  • currentCoverage, nextStep, and gaps text fields
  • Automation paths, API refs, interaction refs, and tickets
Prompt intent

What the prompt should ask

Describe readiness from actual evidence, not aspiration. Highlight the weakest evidence surface and propose the next verification loop in a way that respects the section's existing proof boundaries.

Proof Type proof-ladder

Proof Ladder should emphasize staged proof truth, not generic progress.

This type is ordered. The AI layer has to respect that ordering and treat each rung as an increase in proof strength. The most useful advisory outputs are the current closure state, the weakest unsatisfied rung, and the next stronger proof step.

proof-state-brief

Summarize what the ladder currently proves, up to which rung, and what that proof still does not justify.

Top-of-section proof summary

weakest-open-rung

Identify the first rung that is not actually satisfied and explain what evidence is missing for closure.

Inline rung callout

next-stronger-proof

Generate a concise statement of the next evidence shape that would increase proof strength without redesigning the ladder.

Action card at ladder footer
Prompt ingredients

What the renderer should feed

  • promptGuidance for staged escalation and ladder ordering
  • Rung order as it appears in the section
  • Rung status and notes
  • Any ticket refs or evidence notes linked to each rung
Prompt intent

What the prompt should ask

State what the current ladder genuinely proves, identify the first unsatisfied rung, and describe the next stronger proof move without flattening the ladder into a progress checklist.

Output contract

What JSON should come back

  "currentProofBoundary": {

    "summary": "string",

  },
  "weakestOpenRung": {
    "rung": "string",
    "missingEvidence": "string"
  },
  "nextStrongerProof": {
    "targetRung": "string",
    "requiredEvidence": ["string"]
  }
}

Proof Ladder AI advisory

Proof state brief

The ladder currently supports replayable evidence through the semantic equivalence rung, but it does not yet justify live production drift claims.

Weakest open rung

The first open rung is production drift detection because no same-shape operational evidence has been collected under live conditions.

Next stronger proof

Collect one live drift case using the current ladder shape, then replay and verify it without introducing a new harness or changing the rung model.

Cross-Type UI Rules

The export UX should use the same slot grammar across all three types.

The content changes by type, but the placement language should stay recognizable so readers know these are advisory overlays, not canonical content.

Slot
Purpose
Rule
section-brief
Top-of-section compression block for the current type.
Always render above the section data grid and mark clearly as local AI advisory.
risk-or-weakness-note
One inline callout for the most important current limitation.
Render inside the section body, but visually distinct from canonical callouts.
next-loop
Short action-oriented list for review or proof progression.
Render near the footer or below the brief, never mixed into the canonical cards.
UI principle: every AI region should be collapsible, clearly labeled as advisory, and safe to ignore. The document must remain fully usable if hydration never occurs.
Renderer Contract

The first implementation only needs one export pattern.

The renderer should inspect the section's convergence type, resolve enabled profiles, emit the matching data-ai-source and data-ai-task slots, embed the validated spec, and stop there. No authoring UI changes are required for the first pass beyond making these outputs legible in exported HTML.
