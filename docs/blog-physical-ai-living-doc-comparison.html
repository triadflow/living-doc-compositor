Comparative analysis Issue 01 · № 06

Two ways of making complexity legible to AI.

Diode and Unlimited turn physical design into constrained code-like systems. The inference-time living-doc harness turns agent work into a governed document substrate. The common move is not automation first. It is representation first.

The a16z Physical AI episode is easy to file under robotics, construction automation, or industrial AI. That is not the most useful reading. The sharper pattern is that both companies described in the transcript attack complexity by changing the substrate on which the model works.

Diode does not ask a frontier model to become an expert PCB designer in the raw. Unlimited does not ask a model to improvise a construction project from loose natural language. Both build rails: compilers, ontologies, validated blocks, deterministic checks, simulation, and end-to-end delivery surfaces. Living Doc Compositor makes a parallel move for inference-time work. It does not ask Codex to keep an entire complex objective in free-form context. It turns the work into a governed living document with typed sections, facets, coverage edges, invariants, readiness gates, and source-linked evidence.

Core comparison

Physical AI companies make physical work AI-addressable by converting it into constrained representations. Living Doc Compositor makes complex agent work AI-addressable by converting it into governed representations.

Source Frame

The transcript centers on two founders: Davide Asnaghi of Diode Computers, focused on AI-designed and manufactured PCBs, and Alex Modon of Unlimited Industries, focused on AI-native design and EPC for large infrastructure. The repeated claim is that the breakthrough is not simply better prompting. It is putting the physical domain onto rails so frontier models can operate in a shape they are already good at.

The comparison point on our side is inference-living-doc-run-codex. That skill creates a fresh governed run document for a complex objective, uses the living-doc registry as a type system, tracks documentReady and objectiveReady separately, requires source-system evidence before status upgrades, and forces Codex back to objective work after each harness cycle.

Diode

PCB complexity as compilable design intent.

Diode reframes board design as a code-like target. The model expresses intent through a compiler-like layer, then deterministic design-for-manufacturing checks and validation loops carry the burden of physical correctness.

Unlimited

Infrastructure complexity as an ontology.

Unlimited appears to model projects as structured systems with dependencies, deliverables, schedules, and physical constraints. The target is not a CAD plugin. It is the finished IFC/EPC output.

Living Docs

The harness turns a complex Codex objective into typed sections, facets, evidence, governance, and readiness gates. The model still executes, but the doc makes execution auditable.

The Shared Move

The shared move is representation before automation. Each system chooses a substrate where model output can be constrained, checked, and accumulated over time.

Problem Physical AI answer Living-doc answer
Raw domain is too complex for free-form generation. Do not generate the physical artifact directly. Generate structured design intent that compilers, ontologies, and validators can process. Do not solve the whole objective from chat memory. Generate and update typed run state that governance and coverage can inspect.
Correctness cannot be trusted from prose. Use design rules, manufacturability checks, simulation, and physical validation. Use source anchors, evidence links, governance evaluation, rendered docs, commits, tests, PRs, and issue updates.

Learning data is scarce or scattered. Bootstrap from validated blocks, deterministic rails, simulation, and repeated delivery loops that generate new training data. Bootstrap from source systems, typed cards, coverage edges, invariants, and run history that generate reusable operational memory.
Drift is hard to see. Physical constraints, DFM checks, and simulation expose design drift before manufacture. Governance exposes status drift, source-boundary drift, coverage gaps, and false completion.

Diode: Complexity As Manufacturing-Checked Code

Diode's distinctive move is to make PCB design feel more like writing code than dragging traces in a schematic tool. The transcript summary says Diode built a compiler so the model feels like it is writing Python rather than a schematic. That matters because frontier models are much stronger at code-shaped symbolic work than at raw spatial, electrical, and manufacturing reasoning.

The company is therefore not betting on a model spontaneously possessing tacit DFM judgment. It is betting that enough of the work can be re-expressed as structured intent, and that the missing physical judgment can be supplied by automated checks, manufacturing feedback, and a growing library of validated design blocks.

Compiler layer The model writes into an intermediate representation where the system can catch invalid outputs before they become expensive.
Validated block library Reusable design blocks turn one successful physical design into future substrate, not just a one-off artifact.
DFM as verification Design-for-manufacturing checks serve the same role tests serve in software: they make hidden constraints executable.
Data flywheel Every manufactured or validated design can become training material, examples, or rails for the next design.

This is close to how a living-doc run treats complex Codex work. We do not let the agent's plan remain a blob of text. We move it into typed carriers: objectives, facets, source anchors, status fields, verification evidence, governance invariants. The model can still reason, but its reasoning has a place to land.

Unlimited: Complexity As Deliverable-Centered EPC

Unlimited's approach, as distilled from the transcript, is less about a single technical compiler and more about changing the business and workflow boundary. The company sells the output, not the software. That matters because construction and EPC markets have severe adoption friction. Project finance wants predictable internal rates of return and stable delivery risk. A new tool that promises upside but injects process risk is hard to adopt from below.

So Unlimited's move appears to be vertical integration around the deliverable: produce the package, compress schedule, and absorb the complexity inside an AI-native delivery process. The customer does not need to adopt a new AI toolchain. The customer gets a finished project artifact.

Unlimited pattern What it solves Living-doc analogue
Sell the deliverable, not the tool. Bypasses conservative adoption loops in high-risk physical industries. The harness must deliver source-system outcomes. A complete doc with incomplete objective is failure.
Time is the value metric. Engineering cost is often small relative to schedule compression and financeability. The harness exists to reduce objective drift and rework across long-running agent tasks.
Autonomy from day one. Human-in-the-loop assumptions produce a different architecture than full automation assumptions. The skill creates a full governed run from the start rather than adding audit state after the work is already messy.
Ontology over loose workflow. Infrastructure work needs explicit dependencies, deliverables, constraints, and interfaces. Living docs use registry-defined convergence types, facets, source cards, and invariants.

The strongest parallel is the refusal to treat AI as a sidecar. Unlimited does not appear to be adding AI autocomplete to old EPC. The living-doc harness similarly does not add a nicer checklist to ordinary chat. It changes the operating substrate of the work.

The Living-Doc Harness: Complexity As Governed Inference

The inference-time living-doc harness is built around a different failure mode. Physical AI systems are fighting physics, fabrication, scheduling, tacit knowledge, and institutional adoption. Codex is fighting context loss, premature completion, source ownership mistakes, hidden assumptions, status drift, and verification gaps.

The harness turns those failure modes into explicit gates:

  • Fresh objective-specific run doc: no inherited stale plan. Every complex task gets a fresh governed state.
  • Registry-defined structure: convergence types prevent sections from becoming arbitrary buckets.
  • Objective facets: the work is decomposed into things that can be represented, worked, and verified.
  • Coverage edges: each facet must have carriers in the doc rather than living only in the agent's head.
  • Governance invariants: local rules prevent false completion, source-boundary drift, and unverified status upgrades.
  • Readiness split: documentReady and objectiveReady are separate so a clean document cannot masquerade as finished work.
  • Momentum rule: after one pure harness cycle, Codex must return to objective work.
  • Source-system delivery: code, PRs, issues, docs, tests, and verification evidence remain canonical.

That last point is important. The living doc is not the deliverable unless the objective is to create or update a living doc. It is working memory, governance, and audit state. This is similar to Diode's compiler or Unlimited's ontology: the representation is not valuable because it looks complete. It is valuable because it improves delivery of the real artifact.

Where The Approaches Differ

The analogy is strong, but it is not identity. The differences matter because they show where each system has to be strict.

Dimension Diode / Unlimited Living-doc harness
Primary reality constraint Physics, manufacturability, construction sequence, materials, trades, safety, and finance. Source truth, repo truth, issue truth, test truth, PR state, and verified objective completion.
Failure cost Bad board, bad construction package, rework, schedule slip, physical waste, safety risk. Wrong code, false status, unreviewed PR, missing verification, misleading doc, unresolved issue.
Validator Design rules, simulation, physical manufacturing, field constraints. Governance tools, coverage checks, tests, diffs, commits, PRs, issue updates, rendered docs.
Customer-facing artifact Board, manufactured part, IFC package, completed infrastructure work. Implemented code, updated source system, durable design note, issue/PR state, verified artifact.
Accumulation mechanism Validated blocks, generated data, manufacturing feedback, simulation traces. Run docs, source cards, invariants, commits, issues, PR links, verification evidence.
Human role Move from operator of tools toward specifier, reviewer, manufacturer, and business owner. Move from context re-explainer toward objective setter, reviewer, and source-system owner.

The Key Design Lesson

The main design lesson is that AI systems become useful in complex domains when they stop relying on raw conversational intelligence as the only control surface. The transcript's companies do not trust the model alone. They give it a domain-specific substrate where intermediate outputs can be inspected and corrected. The living-doc harness does the same for Codex.

The model is not the system. The system is the model plus the representation, validators, feedback loops, and delivery boundary around it.

This is why the comparison matters for Living Doc Compositor. The harness should not be framed as documentation overhead. It is closer to a compiler and validation layer for agent work. Its job is to make an otherwise fragile inference process executable, inspectable, and repeatable.

What This Suggests For Our Work

The transcript suggests four useful refinements to how we talk about and build the living-doc method.

  1. Use the phrase complexity substrate deliberately. It captures the thing Diode, Unlimited, and Living Doc Compositor share: a constrained layer where AI work becomes checkable.
  2. Emphasize deliverable-first design. The harness exists to complete objectives, not to produce beautiful run docs. That matches Unlimited's sell-the-output pattern.
  3. Make validators more first-class. In physical AI, validation is obvious because physics punishes mistakes. In agent work, validators are governance, tests, diffs, PR state, and issue state. They need the same prominence.
  4. Treat typed source material as the data flywheel. Just as validated blocks become future design substrate, well-structured run docs and source-linked cards become future agent substrate.
Potential essay thesis

The next wave is not only bigger models for harder domains. It is better substrates around models: compilers for PCB design, ontologies for infrastructure, governed living docs for agent work. The winning systems convert complexity into a representation where AI can act and where reality can push back.

Comparison In One Sentence

Diode and Unlimited make physical complexity tractable by turning it into constrained design systems; Living Doc Compositor makes inference-time work tractable by turning it into a governed, source-linked, verifiable run system.
