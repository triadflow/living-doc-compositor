Repair Skills As Independent Inference

The controller does not repair the living doc. It starts accountable inference runs that inspect evidence, change the doc, and leave their own proof.

The practical pattern: worker inference does the work, reviewer inference inspects the worker log, balance-scan inference orders the repair skills, and each repair skill runs as its own headless inference before the worker loop continues.

Worker

Does objective work against the living doc and source files. Its final message is evidence, not authority.

Reviewer

Opens the worker raw JSONL logs by path and emits the authoritative stop signal.

Repair Skills

Run as separate inference sessions. They patch and render the living doc before the next worker run.

Normal Loop

does work"]
  B --> C["Worker raw JSONL log"]
  C --> D["Reviewer inference

  D --> E{"Stop verdict"}
  E -->|closed| F["Lifecycle stops
proof accepted"]
  E -->|resumable| G["Next worker iteration"]
  E -->|repairable| H["Balance-scan inference"]
  H --> I["Ordered repair skill list"]
  I --> J["Repair skill inference runs"]
  J --> K["Living doc patched + rendered"]
  K --> D

The important handoff is not a summary. It is file paths: living doc JSON, reviewer verdict, handover, and raw worker JSONL logs.

Balance Scan Chooses The Repair Order

  S["living-doc-balance-scan
independent inference"] --> T{"Imbalance type"}
  T -->|equilibrium-shift| A["equilibrium-rebalance"]
  T -->|conservation-leak| B["objective-conservation-audit"]
  T -->|illegal-transition| C["reaction-path-validator"]
  T -->|activation-shortfall| D["activation-energy-review"]
  T -->|compound-instability| E["catalytic-repair-run"]
  T -->|fresh-but-open| F["continue worker loop"]
  A --> R["ordered repair plan"]
  B --> R
  C --> R
  D --> R
  E --> R
  F --> R

The scan is a triage inference. It should not quietly repair. It says which repair skill should run first, second, third, and why.

Each Repair Skill Is Its Own Actor

sequenceDiagram
  participant L as Lifecycle
  participant S as Repair Skill Inference
  participant D as Living Doc
  participant W as Worker Logs
  L->>S: skill name + doc path + verdict path + raw JSONL paths
  S->>D: read current doc
  S->>W: inspect worker/reviewer evidence
  S->>D: patch doc
  S->>D: render HTML
  S-->>L: result JSON + changed files + own JSONL log

Minimum Result Contract

  • skill: the skill that ran
  • status: repaired, no-op, blocked, failed
  • changedFiles: concrete paths
  • renderedHtmlPath: required when doc changed
  • nextRecommendedAction: continue, run next skill, block
  • rawJsonlLogPath: the repair inference log

Short Version

The missing system part is not smarter routing. It is execution: when a repair skill is ordered, the lifecycle must start a new headless inference run for that skill, pass it the real evidence paths, let it repair the living doc, and only then continue the worker loop.
