The skill before the living doc can execute.

Many living docs begin inside an ongoing Codex or Claude Code session. The first objective is often directionally right, but too general to close against. That is where objective-acceptance-shaping belongs.

Rene Luijk Triadflow
DraftedMay 7, 2026 Read3 min StatusDraft

The first version of a living-doc objective is rarely bad. It is usually too soft. It captures a direction, a concern, or a desired improvement, but it does not yet say what must become true, what would prove it, and what is outside the work.

That softness matters because the objective becomes the pressure source for the whole living doc. If it is broad, the acceptance criteria become polite. If it is vague, implementation can choose a convenient slice. If it mixes motivation, approach, and outcome, a later agent has to guess which part is the actual contract.

The job of objective-acceptance-shaping: turn rough session intent into one accountable objective, then attach acceptance criteria that can fail against source-system proof.

The early-session problem

Living docs often start while work is already moving. A user and an agent are exploring a refactor, a research direction, a product surface, or a repair. The conversation produces useful language, but not yet a closure contract.

This is exactly the dangerous moment. If the doc is created from that language too quickly, it can look mature while still carrying an objective that no later agent can execute literally.

What the skill does

Critiques the rough objective Finds where it is too broad, abstract, activity-shaped, or detached from source-system proof.
Separates the layers Pulls apart motivation, objective, approach, evidence, and scope boundary so they stop masking each other.
Names accountable terms Finds the nouns, verbs, and standards that cannot disappear during implementation.
Drafts acceptance criteria Creates criteria tied to objective terms, each with proof that can be checked later.

The sequence

  1. objective-acceptance-shapingMake the first objective and acceptance criteria accountable.
  2. create or update the living docPut the objective, criteria, scope, and convergence types into the document.
  3. objective-execution-readinessCheck whether the document can now drive work without drifting.
  4. implementation skillsDo the work against the objective-bound gates.
  5. repair or closure skillsRebalance, audit, or close only when the proof conserves the objective.

This order is important. Readiness is not a substitute for shaping. A readiness skill can say the doc is not executable, but the shaping skill helps create the objective that makes execution possible.

The practical value

The skill gives the user and agent a shared first contract before the living doc hardens around weak wording. It makes the objective less decorative and the acceptance criteria less ceremonial.

That is the small but important shift: the living doc does not start as a nice summary of the session. It starts as a surface that a future agent can obey, challenge, and close against.
