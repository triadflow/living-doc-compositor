Field note Issue 01 · № 07

The run skill is starting to behave like a small delivery system.

Early evidence from the video-intake console: after an epic and GitHub issues gave the work a source-system spine, Codex 5.5 high handled much of the small-to-mid-sized implementation loop through the living-doc harness.

RL

This is not a general benchmark claim. It is a field note from one concrete build: the local video-intake console now running at http://127.0.0.1:4177/. Roughly speaking, the first 80 percent of the app came together cleanly once the work had an epic, issue breakdown, and an inference-time living-doc run as the operating spine.

The important part is not that Codex wrote code. That is the expected part. The interesting part is that the run skill made the work feel less like a long chat and more like a small delivery system: source issue first, governed run doc second, implementation third, verification and PR state kept in view throughout.

Scope of the claim

The claim here is narrow: on small and mid-sized implementation tasks, with Codex 5.5 high, the A-to-Z run skill appears to improve completion discipline when there is already a GitHub issue structure to work from. It does not prove autonomous delivery for large systems, ambiguous product strategy, or tasks with weak verification.

The Case

The project was a new local app: a browser GUI for watching YouTube videos, queueing public videos and playlists, writing watch notes, selecting local Claude Code skills, and running background processing through the local CLI rather than an API. The user pain point was concrete: while watching potential source material, it should be easy to defer the decision of what pipeline the video belongs in.

The work began with GitHub structure rather than code. Issue #1 became the epic. Sub-issues covered app shell, durable queue state, skill discovery, local Claude Code calls, YouTube playlist import, processing history, living-doc-compositor integration, removal from queue, serial job processing, playlist subscriptions, and dismissed-video tracking.

Epic first Issue #1 defined the end-to-end video intake console rather than letting the implementation invent its own target.
PR as delivery surface PR #9 carries the working app, verification notes, and remaining gaps.
Run doc as state The PR references docs/living-runs/video-intake-console-issue-1-run.{json,html} as the governed run record.

What The App Reached

By the time the first loop stabilized, the app had a local Node server, a static GUI, durable queue and job storage under .video-intake-data/, YouTube video import, playlist import, separate watch and processing states, queue filters, per-video notes, processing history, artifacts, configured skill actions, and a local claude -p runner with stdout, exit, duration, timeout, and downstream status capture.

Later iterations added the pieces that made the tool feel real rather than demo-shaped: serial background processing, playlist subscriptions, dismissed-video memory, refined skill action cards, autosaved watch notes, optional extra directions, real skill-path resolution, Claude session-log linkage, and a copy action for queue-item metadata.

Queue substrate
.video-intake-data/state.json stores queue items, notes, playlist subscriptions, dismissed videos, job state, and history.
Local skill runner
The GUI starts local Claude Code jobs through configured actions, not through an Anthropic or OpenAI API.
Processing discipline
Background work is serialized so one video-processing job finishes before the next one starts.
Audit surface
Each queue item can expose recent jobs, run logs, artifacts, status, and Claude session JSONL paths.

Why The Skill Helped

The A-to-Z run skill helped because it forced the work through a delivery shape. It did not rely on a single long chat remembering every decision. The harness required scope, source anchors, expected outputs, verification evidence, and a distinction between a coherent run document and an actually completed objective.

01

GitHub issue gives the work a canonical objective.

02

Living run doc decomposes the objective into facets and evidence carriers.

03

Codex implements against the repo while the harness tracks status truth.

04

Verification, PR state, and residual gaps become part of the delivery record.

The useful behavior was not magic. It was pressure. The skill keeps asking: what is the objective, which source system owns the truth, what changed, what was verified, what is still partial, and what would make the work objectively ready? That pressure is exactly what normal agent sessions often lose after the third or fourth implementation turn.

What It Did Not Solve

The skill did not eliminate judgment. The GUI still needed human taste. Skill action names still needed correction. The difference between a dossier-scan action and an /integrate-source action needed domain knowledge. The transcript showed cases where source assessment skipped a relevant inference-economics angle until the skill itself was sharpened.

That is an important boundary. The harness improves execution discipline; it does not make the model immune to wrong salience, weak product judgment, or missing domain distinctions. Its value is that those mistakes become inspectable and repairable rather than disappearing inside a chat transcript.

Worked well Still needs human correction
Turning an epic and sub-issues into a working local app skeleton. Deciding which skills are semantically real product actions.
Maintaining queue/job persistence and app behavior across iterations. Recognizing when a source's strategic claim belongs in a different dossier.
Capturing logs, job state, session paths, and verification trails. Judging UX refinement: density, labels, affordances, and what should be visible.
Keeping work tied to PR #9 and the issue structure. Closing the final 20 percent: polish, edge cases, review, and publication discipline.

The Pattern

The emerging pattern is simple: the run skill works best when the world has already been made legible. An epic gives Codex a target. Sub-issues give it boundaries. The living run doc gives it state. The PR gives it a delivery surface. Tests and local checks give it evidence. With those in place, Codex 5.5 high can carry a surprising amount of the implementation loop.

Without those pieces, the skill would still create structure, but it would spend more of the run discovering the shape of the work. The video-intake console worked because the project started with a source-system spine.

Working hypothesis

For small and mid-sized build tasks, the strongest setup is: write the epic, split the issues, then let the A-to-Z run skill operate through the repo, PR, and living run doc. The model is still doing the work, but the source systems keep the work from dissolving into conversation.
