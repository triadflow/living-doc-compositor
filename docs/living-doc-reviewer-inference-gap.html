Agentic living-doc harness

The missing layer was reviewer inference.

The harness could run workers, collect artifacts, inspect logs, and move to another iteration. The gap was that the close-or-continue signal was still being created mostly by deterministic proxy checks. That misses the core design: inference must judge the run state, then deterministic code must enforce the transition.

Plain statement

  • The worker inference does the objective work.
  • A separate reviewer inference must judge what the worker state means.
  • The deterministic lifecycle must only validate, record, and execute allowed transitions.
  • Closure is invalid if it comes only from fields like objectiveReady: true, completed criteria, and rendered HTML.
current gap

What was lacking

The implementation had a good artifact spine, but the meaning-making step was in the wrong place.

What the system drifted into

After the worker stopped, deterministic code looked at structured state and inferred enough to close or continue.

  • objectiveReady === true
  • acceptance criteria all complete
  • rendered HTML exists
  • native trace refs exist
  • proof gates pass
Why this is not enough

Those fields can support a verdict, but they should not be the verdict. They are evidence. They do not replace reasoning over why the worker stopped and whether the living-doc objective is actually conserved.

What the objective requires

After the worker stops, a separate reviewer inference reads the evidence and emits a structured verdict. Deterministic code then enforces that verdict.

  • worker run evidence
  • native inference trace summaries
  • living doc before and after
  • source artifacts and changed files
  • acceptance criteria and proof gates
  • worker final message as evidence, not authority
The corrected boundary

Inference produces the meaning signal. Deterministic code enforces protocol, schemas, hard gates, artifact writing, and legal transitions.

correct chain

The input-output chain between iterations

A normal run should have two inference roles: the worker that acts, and the reviewer that judges the stop state.

1

Worker input

The lifecycle controller starts from the living doc objective, current stage, acceptance criteria, source paths, and any previous output-input handover.

2

Worker inference

Headless Codex performs objective work: edits files, updates the living doc, renders HTML, runs checks, or reaches a blocker.

3

Evidence bundle

The harness freezes contracts, state, event logs, changed files, native trace refs, worker output, proof gates, and living-doc before/after snapshots.

4

Reviewer inference

A separate inference pass reads the evidence and emits the authoritative structured verdict: close, repair, resume, block, pivot, defer, or exhaust.

5

Deterministic enforcement

The lifecycle validates schema and hard gates, writes the verdict reference, then either stops or creates the next output-input artifact.

why it matters

Why deterministic proxy closure is not a harmless shortcut

The living-doc system exists to reduce drift toward partial completion. If the reviewer signal is replaced by code checks, the system repeats the same drift at a lower layer.

It confuses evidence with judgment

Completed criteria and rendered files are evidence. They still need interpretation against the actual objective, invariants, stage, trace, and worker behavior.

It can close on shape

A deterministic check can see a closed-looking document without understanding whether the work conserved the accountable objective terms.

It weakens the harness premise

The premise is that inference can inspect ambiguous stops and repair living-doc balance. If code decides meaning, the harness becomes a runner with gates, not a reasoning loop.

contract handovers

What contracts need to exist

The reviewer inference layer needs its own durable input, output, trace, and lifecycle reference. Otherwise the verdict cannot be inspected later.

Reviewer input contract

Names the worker run, living doc path, objective hash, changed artifacts, native trace summaries, worker final message reference, proof gates, and prior output-input handover. This is what reviewer inference reads.

Reviewer verdict artifact

Contains classification, reasonCode, closureAllowed, evidence citations, unresolved terms, confidence, and the required deterministic action.

Reviewer inference trace

Records the inference pass or sanitized trace reference proving the verdict came from an inference review, not from lifecycle code manufacturing the signal.

Lifecycle result reference

The lifecycle result must point to the reviewer verdict used for each transition. A terminal state without that reference is not valid closure.

Dashboard surface

The dashboard must show the reviewer verdict next to hard gates, worker state, native traces, blocker state, and output-input paths so a later reviewer can see why the loop stopped or continued.

artifact map

Where the new evidence sits

The reviewer layer does not replace existing artifacts. It sits between worker evidence and lifecycle transition.

Sanitized hard evidence of what the headless worker actually did.

Reviewer input

The frozen evidence packet sent into the reviewer inference pass.

.living-doc-runs/<run-id>/reviewer-inference/iteration-N-input.json

Reviewer verdict

The structured meaning signal produced by inference.

.living-doc-runs/<run-id>/reviewer-inference/iteration-N-verdict.json

Terminal or output-input

The deterministic lifecycle action after validating the reviewer verdict.

.living-doc-runs/<run-id>/terminal/iteration-N-closed.json

Lifecycle result

The full chain showing worker iteration, reviewer verdict, transition, and final state.

.living-doc-runs/ldhl-.../lifecycle-result.json
acceptance tests

Tests that make the gap real

The corrected objective needs tests that fail on the current shortcut.

No reviewer, no closure

A fixture with objectiveReady: true, all criteria complete, rendered HTML present, and native trace refs still cannot close unless a reviewer-inference verdict exists and is referenced.

Two-phase calibration requires reviewer verdicts

Iteration 1 must have a reviewer verdict that keeps the loop going. Iteration 2 must have a reviewer verdict that allows closure. The lifecycle result must reference both.

Reviewer can disagree with worker

If the worker says done but proof is incomplete, the reviewer emits closure-candidate or repairable, and deterministic code must not close.

Hard gates can reject reviewer output

If reviewer inference emits malformed JSON, cites missing artifacts, or allows closure while required hard gates fail, deterministic code rejects the transition and records the defect.

current status

The honest status after the correction

The live two-iteration run remains valuable, but it does not satisfy the corrected objective by itself.

The key correction

The system is not complete when deterministic code can derive that a run looks closed. It is complete when reviewer inference produces the stop signal from evidence, and deterministic code can only enforce or reject that signal according to contracts.
