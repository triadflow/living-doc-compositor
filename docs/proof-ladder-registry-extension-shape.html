Convergence Type Proposal

Proof Ladder Registry Extension Shape

Make ladder prominence native to the convergence type system by teaching proof-ladder how to derive frontier, closure, inversion, and summary-worthy signals from its own ordered state.

Proposal only No live registry patch yet Centered on proof-ladder
Problem

Current registry shape stops too early

The current type defines ordering and status, which is enough to render a ladder section. It does not define canonical derived state for UI or inference.

  • No type-level definition of the current frontier
  • No contiguous closed-prefix logic
  • No inversion rule beyond one AI action
  • No type-native summary eligibility for hero or snapshot surfaces
Move

Teach the convergence type its own semantics

Add two optional fields on convergence types:

  • stateSemantics — how ordered state is interpreted and derived
  • summarySemantics — which derived signals deserve prominence
source of truth stays on the type summary surfaces consume, not invent
Before / After

What changes in the system boundary

Before

  • proof-ladder says “ordered probe section”
  • Consumers infer frontier logic ad hoc
  • Hero or snapshot prominence depends on UI heuristics
  • Meaning leaks into renderer or prompts

After

  • proof-ladder declares how ladder state derives
  • Frontier and closed-prefix become canonical signals
  • Prominence becomes eligible through summarySemantics
  • Consumers read the type instead of hard-coding rules
Exact Shape

Registry extension

{
  "stateSemantics": {
    "kind": "ordered-ladder",
    "orderedBy": "data-order",
    "itemUnit": "rung",
    "statusKey": "status",
    "closedStatuses": ["ready"],
    "activeStatuses": ["partial", "blocked"],
    "futureStatuses": ["planned"],
    "frontierRule": {
      "kind": "first-status-not-in",
      "statuses": ["ready"]
    },
    "closedPrefixRule": {
      "kind": "contiguous-prefix-in-statuses",
      "statuses": ["ready"]
    },
    "inversionRule": {
      "kind": "status-after-open-prefix",
      "openStatuses": ["partial", "planned", "blocked"],
      "invalidLaterStatuses": ["ready"]
    },
    "derivedSignals": [
      { "id": "closedPrefixCount", "kind": "contiguous-prefix-count" },
      { "id": "currentFrontierItem", "kind": "first-item-with-status-not-in" },
      { "id": "currentFrontierStatus", "kind": "status-of-signal" },
      { "id": "nextRungItem", "kind": "next-item-after-signal" },
      { "id": "isFullyClosed", "kind": "all-items-in-statuses" },
      { "id": "hasInversion", "kind": "violates-rule" }
    ]
  },
  "summarySemantics": {
    "headlineSignal": "currentFrontierItem",
    "prominentSignals": [
      "currentFrontierItem",
      "currentFrontierStatus",
      "closedPrefixCount",
      "nextRungItem"
    ]
  }
}
Interpretation Flow

How the type now behaves

Input section

Ordered rung items The section’s data array remains the canonical ordered ladder.
Rung statuses Status continues to use the existing probe-status set: planned, partial, ready, blocked.
→

Derived by the type

Frontier First item not in ready.
Closed prefix Contiguous leading run of ready items.
Inversion Any later ready after an open prefix.
→

Consumed by surfaces

Hero / top summary Can prominently surface the frontier and closed count.
Status snapshot Can compress ladder state without inventing ladder logic.
Flow / integrity views Can highlight inversion or invalid closure patterns.
Current Doc Example

What this derives on the current Proof Ladder Overview

The proposal matches the current live ladder state exactly. These values become type-native derived signals rather than UI guesses.

Closed Prefix Count 5 Levels 1–5 are contiguous ready.
Current Frontier Level 6 First rung not in ready.
Frontier Status partial Autonomous shared-runtime repair and resume remains open.
Next Rung Level 7 Repeat later without redesign stays behind the current frontier.
Fully Closed false The ladder is not complete.
Has Inversion false No later ready rung appears after an open rung.
Architectural Effect

Meaning stays attached to the type

The convergence type system becomes a registry of structural meaning and derivation rules, not only a rendering registry.

Consumer Effect

Summary surfaces stop guessing

status-snapshot, hero summaries, and integrity views can consume derived ladder signals without re-implementing ladder semantics.

Generalization

Reusable pattern

If this holds up, stateSemantics and summarySemantics can extend other ordered or stateful convergence types later.

Next Step

Decision points before patching the registry

  • Decide whether the compositor consumes these directly or exposes them to AI/sync layers first.
  • Then patch scripts/living-doc-registry.json with proof-ladder as the first adopter.
